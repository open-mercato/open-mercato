# Combine Dependabot PRs #2064 and #2062 onto develop

## Overview

Goal: consolidate Dependabot PRs `#2064` (major group, 4 packages) and `#2062` (minor-and-patch group, 28 packages) into a single replacement PR targeted at `develop`, then close the originals (which target `main`).

Affected area: dependency manifests and lockfile only.

Source PRs:

- `#2064` — `chore(deps): bump the major group with 4 updates` (glob, testcontainers, undici, react-is)
- `#2062` — `chore(deps): bump the minor-and-patch group with 28 updates` (mikro-orm 7.0.17→7.1.1, esbuild 0.25→0.28, ai-sdk, aws-sdk, jest, framer-motion, etc.)

Both target `main`; user wants a single PR against `develop`. Precedent: PR `#2005` (slug `migrate-dependabot-deps-to-develop`).

## Scope

- Branch off `origin/develop` as `fix/combine-deps-prs-2064-2062`.
- Apply the package.json changes from both Dependabot branches.
- Regenerate `yarn.lock` from the merged manifest set.
- Validate: install, generate, build packages, typecheck, run unit tests, build app.
- Open one replacement PR against `develop` with `dependencies` label.
- Close `#2064` and `#2062` with supersession comments pointing at the new PR.

## Non-goals

- No runtime code or generated-output changes beyond what the dependency upgrades require.
- No additional package bumps outside the two source PRs.
- No engagement with the `undici` / `react-is` bumps from `#2064` — those have already been merged onto `develop` (`packages/shared/package.json` is at `^8.3.0`; `packages/ui/package.json` is at `^19.2.6`). Only `glob` and `testcontainers` from `#2064` still need to land on `develop`.

## Risks

- Lockfile shape will differ from both source PRs (develop has moved). Mitigated by full validation gate.
- `esbuild` 0.25→0.28 and `testcontainers` 11→12 are the largest semver jumps; build + integration test failures would surface here.
- `glob` 11→13 changes ESM/typing surface — only used in `packages/storage-s3` devDependencies, so blast radius is the storage-s3 build.
- Mikro-ORM 7.0.17→7.1.1 is a minor; per-tenant queries are heavily covered by unit tests, so any regression should surface during `yarn test`.
