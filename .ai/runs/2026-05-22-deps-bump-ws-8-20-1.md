1. Run `yarn up -R ws@8.20.1` to recursively pin every `ws@^8.17.1 / ^8.18.0` consumer to `8.20.1` (the same shape as the upstream Dependabot diff). 2. Confirm the resulting diff is limited to `yarn.lock` and that the `"ws@npm:^8.17.1, ws@npm:^8.18.0"` block now points at `8.20.1` with a fresh checksum.
3. Run `yarn install --immutable=false` (default) to make sure yarn accepts the lockfile, then a follow-up `yarn install --immutable` to prove the lockfile is internally consistent.

## Risks

- **Lockfile churn beyond the targeted group**: `yarn up -R ws@8.20.1` could pull in unrelated dedupe changes. Mitigation: re-read the diff after running yarn; revert and fall back to a manual lockfile edit if anything outside the `^8.17.1, ^8.18.0` group changes.
- **Indirect breakage in transitive consumers**: ws 8.20.1 contains a tiny patch fix only (CVE for `websocket.close()`); the API surface is identical. Mitigation: rely on `yarn typecheck` and `yarn test` to surface breakage.
- **Source PR was closed without merging**: maintainers may have closed it because Dependabot already opened a follow-up PR or because of policy. Mitigation: explicitly reference PR #2016 in the new PR body so a reviewer can re-confirm the merge decision.
- **No external skill conflicts**: the run did not adopt any external skill rules.
