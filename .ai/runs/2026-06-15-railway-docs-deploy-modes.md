# Railway Deployment Docs Source Mode Clarification

## Overview

Source spec: `.ai/specs/2026-05-12-railway-one-command-deploy.md`

Goal: make the Railway deployment guide explicit about prerequisites, token choice, Git-backed versus local-source modes, Railway CLI installation, account tier guidance, environment-file preparation, and the `RAILWAY_API_TOKEN` login pitfall.

Affected files:

- `apps/docs/docs/deployment/railway.mdx`

External References:

- https://docs.railway.com/cli - adopted the current Railway CLI install and token environment variable guidance; rejected none.
- https://docs.railway.com/cli/login - adopted the warning that `RAILWAY_API_TOKEN` or `RAILWAY_TOKEN` makes `railway login` skip interactive login and that only one should be set at a time; rejected none.
- https://docs.railway.com/integrations/api - adopted Railway's distinction between Account, Workspace, and Project tokens and the "No workspace" account-token option; rejected none.
- https://railway.com/pricing - adopted current resource-limit guidance showing Hobby and Pro both exceed 16 GB RAM per service, with Pro better for production/team deployments; rejected none.

## Scope

- Clarify prerequisites before the quickstart.
- Expand source mode docs with clear Git and local-source paths.
- Add Railway CLI installation guidance only where it is needed for local uploads.
- Explain how to create the recommended account token and how to avoid token-type mistakes.
- Mention the need for a real standalone app `.env` generated by `yarn setup` after `create-mercato-app`.
- Add troubleshooting for `railway login` being blocked by an invalid exported token.

## Non-goals

- No CLI behavior changes.
- No generated file changes.
- No Railway pricing assertions beyond current docs links and high-level plan guidance.
- No live Railway deployment validation.

## Risks

- Railway CLI install commands or pricing limits may change; mitigate by linking to Railway docs and avoiding hard-coded install script bodies where possible.
- Overstating account-tier requirements could make hobby/test users think Pro is mandatory; mitigate by saying a paid plan with enough resources is needed, with Pro recommended for production/team deployments.
- Token examples must never include real token values; keep placeholders only.
