# Port Dependabot Bumps from PRs 1970, 1971, 1972, 1974, and 1975

## Overview

Goal: consolidate the dependency bumps from Dependabot PRs 1970, 1971, 1972, 1974, and 1975 into one PR against `develop`.

Source PRs:

- PR 1970: `chore(deps): bump dompurify from 3.3.3 to 3.4.5`
- PR 1971: `chore(deps): bump postcss from 8.4.31 to 8.5.14`
- PR 1972: `chore(deps): bump ip-address from 10.1.0 to 10.2.0`
- PR 1974: `chore(deps): bump the minor-and-patch group with 28 updates`
- PR 1975: `chore(deps): bump the major group with 8 updates`

## Scope

Affected files are expected to be package manifests and `yarn.lock` only:

- Root `package.json`
- App manifests under `apps/docs` and `apps/mercato`
- Package manifests touched by the source Dependabot PRs
- `yarn.lock`

Non-goals:

- No source code migration unless required by validation failures caused by the dependency updates.
- No module, API, database, event, ACL, or UI behavior changes.
- No changes to unrelated generated files.

## Implementation Plan

### Phase 1: Plan and Branch Setup

Create the isolated branch from `origin/develop`, install dependencies, and commit this run plan so the work is resumable.

### Phase 2: Consolidate Dependency Bumps

Port the manifest changes from the source PRs, regenerate `yarn.lock` through Yarn, and verify the resulting diff is limited to dependency metadata.

### Phase 3: Validation, Review, and PR

Run the dependency-appropriate validation gate, perform code-review and backward-compatibility self-review, open the PR against `develop`, normalize labels, and run the required `auto-review-pr` pass.

## Risks

- Major dependency updates from PR 1975 may expose TypeScript, lint, build, or runtime incompatibilities.
- Combining independent Dependabot PRs may produce a different lockfile than the original per-PR lockfiles; Yarn must be the source of truth.
- If validation requires code changes, those must stay minimal and directly tied to the dependency updates.
