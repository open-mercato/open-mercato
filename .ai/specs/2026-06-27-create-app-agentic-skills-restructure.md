# Standalone Agentic Skills Restructure — Thin SKILL.md + Install-Time Parametrization

- **Status:** Proposed (design complete — not yet implemented)
- **Date:** 2026-06-27
- **Scope:** OSS — `packages/create-app/agentic/shared/ai/skills/**` + the setup wizard/generators
- **Related:** companion spec `.ai/specs/2026-06-27-ts-morph-module-fact-sheets.md` (PR #3685, restructures the *guides* half of `agentic/`); `packages/create-app/src/setup/wizard.ts`, `packages/create-app/src/setup/tools/shared.ts`, `packages/create-app/build.mjs`, `packages/create-app/AGENTS.md`

## TLDR

Restructure all **21** skills under `packages/create-app/agentic/shared/ai/skills/` along a clean separation of concerns:

- **`SKILL.md` = router only** — *when* the skill may be used (triggers, kept in frontmatter `description` so auto-discovery still fires) and *what* it can contain (capability summary), plus a **reference map** to instruction files. The procedure body leaves `SKILL.md`.
- **Instruction files = procedure** — content-descriptive filenames: `instructions.md` for single-procedure skills; `workflow/step-N-<name>.md` for multi-step flows; `subagents/<role>.md` for any subagent a skill spawns (the subagent's full playbook lives in its own file).
- **`.ai/agentic.config.json` = environment** — a single generated config the wizard writes from install-time answers and skills read at runtime. This **replaces the 7 `STANDALONE.md` override files**: the one genuinely per-repo value (PR base branch) becomes an install question stored in config; everything else is authored natively into the instructions or discovered at runtime.

No runtime module behaviour changes — this is agentic-tooling/docs. Domain heuristics (ORM, encryption, DS, undo, tenant scoping) are **N/A** and intentionally not forced.

## Decisions (locked via Open-Questions gate)

| # | Decision | Choice |
|---|----------|--------|
| D1 | How skills consume install answers | **Single runtime config** `.ai/agentic.config.json`; skills are instructed to read it. Not per-file placeholder substitution. |
| D2 | Where the moved body lives + SKILL shape | **Content-descriptive files**: `instructions.md`, or `workflow/step-N-<name>.md` for multi-step, or `subagents/<role>.md` for spawned subagents. Thin `SKILL.md` = frontmatter (`name`, `description`=when-to-use) + "What it contains" + a reference map. Existing `references/<topic>.md` (checklists/catalogs) is retained. |

| D4 | Scope of this spec | **All 21 skills in one pass** (internally phased for reviewability). |

## 1. Problem

1. **Monolithic `SKILL.md`.** Most skills inline the entire procedure: the 7 auto-* skills are **329–749 lines with zero reference links** (`reflinks=0`); progressive disclosure (the Claude Code `references/` pattern) is applied inconsistently across the set (`reflinks` 0–9). A large `SKILL.md` is loaded in full every time the skill triggers, wasting context and burying the trigger/router concern in procedure detail.
2. **`STANDALONE.md` is a fragile parallel override layer.** The standalone `SKILL.md` files were authored from the monorepo and still hard-code monorepo assumptions (e.g. *"base branch is always `develop`"*); `STANDALONE.md` then patches them at runtime and the agent must remember to read it **first**. The same ~6 rules are duplicated across 7 files, and both `wizard.ts` `printSummary` and `create-app/AGENTS.md` still point users at `STANDALONE.md`.

## 2. Architecture — three separated concerns

```
.ai/
├── agentic.config.json              # ENVIRONMENT — generated by wizard, read by skills
└── skills/<skill>/
    ├── SKILL.md                     # ROUTER — frontmatter(when) + what-it-contains + reference map
    ├── instructions.md              # PROCEDURE (single-flow skills)
    ├── workflow/                    # PROCEDURE (multi-step skills)
    │   ├── step-1-<name>.md
    │   └── step-2-<name>.md
    ├── subagents/<role>.md          # PROCEDURE — a spawned subagent's full playbook
    └── references/<topic>.md        # REFERENCE — checklists, catalogs (retained)
```

### 2.1 Thin `SKILL.md` contract (MUST)
A restructured `SKILL.md` MUST contain only:
- **Frontmatter** `name` + `description`. The `description` carries the trigger phrasing (the load-bearing part Claude Code uses to decide invocation) — it MUST NOT be thinned away.
- **When to use** — a short prose/bullet restatement of triggers and non-triggers.
- **What it contains** — a capability summary (what the skill produces/changes).
- **Reference map** — a table listing each instruction file, *when* to load it, and the intended load order (progressive disclosure: load only the step in play).

It MUST NOT contain step-by-step procedure, code blocks longer than a short illustrative snippet, or subagent playbooks — those live in the instruction files.

### 2.2 Config artifact (`.ai/agentic.config.json`)
Generated by the wizard; single source of truth for per-repo environment. Provisional schema:
```json
{
  "projectName": "my-app",
  "agentTools": ["claude-code"],
  "pr": {
    "baseBranch": "main"        // a literal branch, or "auto" → skill resolves via `gh repo view` at runtime
  }
}
```
- Skills that open PRs read `pr.baseBranch`. When it is `"auto"` (or the file/key is absent), the skill falls back to runtime discovery (`gh repo view --json defaultBranchRef`) — the one concession that keeps install-time (no GitHub remote yet) and runtime reconciled, without reintroducing an override file.
- The schema is additive/extensible; new install questions append keys, they do not reshape existing ones.

### 2.3 `STANDALONE.md` → new home mapping
| STANDALONE.md rule | New home | Mechanism |
|---|---|---|
| 1. Base branch (was hard-coded `develop`) | **Install question → config** | `pr.baseBranch`; runtime `gh` fallback when `"auto"`/absent |
| 2. Pipeline labels opt-in | **Native + runtime discovery** | instructions: `gh label list` check, skip-and-log missing, offer `gh label create` |
| 3. Validation-gate script probing | **Native** | instructions: probe `package.json` scripts, run those present |
| 4. `src/modules/` file layout | **Native authoring** | instructions reference standalone paths directly; no monorepo `packages/...` |
| 5. `--skill-url` safety envelope | **Native** | safety rules written into instructions (tool-agnostic) |
| 6. Claim / `in-progress` discipline | **Native** | instructions: assignee + claim comment when `in-progress` label absent |

## 3. Wizard & generator changes

- **`AgenticConfig`** (`wizard.ts`) gains `pr: { baseBranch }` (and stays extensible). When a PR-capable tool is selected, the wizard asks: *"Which branch should automated PRs target? [auto-detect / main / develop / other]"*, defaulting to `auto`. **BC (CLI surface #13):** the non-interactive `--agents`/`--tool` path MUST stay non-interactive — add an **additive optional flag** `--pr-base <branch|auto>` (default `auto`); never make the new question mandatory in headless scaffolds.
- **New generator step** writes `.ai/agentic.config.json` from `AgenticConfig` (in `generateShared` or a dedicated tool).
- **`shared.ts` copy logic** switches from per-skill hard-coded file lists to **recursive copy of each skill directory** — required now that a skill's file tree varies (`workflow/`, `subagents/`, `references/`). The recursive copy MUST run `resolvePlaceholders` on every copied **text** file so `{{PROJECT_NAME}}` keeps resolving (today only the `writeTemplate` files are substituted, e.g. `om-spec-writing/SKILL.md`; raw `copyFile` ones are not — recursive copy must not silently ship `{{PROJECT_NAME}}` literally).
- **`build.mjs` clean step (lessons.md — stale `dist/`):** `cpSync('agentic','dist/agentic',…)` currently does **not** clean first, so the 7 deleted `STANDALONE.md` would linger in `dist/agentic/` and keep shipping via npm/Verdaccio. Add `rm -rf dist/agentic` before the copy and a build assertion that `dist/agentic/**` contains no `STANDALONE.md`.
- **`printSummary`** stops referencing `STANDALONE.md`; mentions the config file instead.
- **Standalone guidance sync (`create-app/AGENTS.md` Always #8):** update `agentic/shared/AGENTS.md.template` (and `template/AGENTS.md` if it documents skill structure) to describe the thin-`SKILL.md` + instruction-files + `agentic.config.json` layout.

## 4. Phasing (all 21 in one spec; staged for review)

1. **Plumbing** — `AgenticConfig.pr.baseBranch`, the install question, `.ai/agentic.config.json` generator, recursive skill-dir copy, remove `STANDALONE.md` mentions from `wizard.ts` + `create-app/AGENTS.md`.
2. **auto-\* (7 skills)** — move bodies to `workflow/` + `subagents/`, absorb the STANDALONE rules natively (§2.3), wire `pr.baseBranch`, delete their `STANDALONE.md`. These are the largest and own the override files, so they set the pattern.
3. **Remaining 14 skills** — thin each `SKILL.md` to the §2.1 contract; relocate bodies to `instructions.md`/`workflow/`/`subagents/` per shape.
4. **Guards + docs** — restructure-conformance tests (§6), update `create-app/AGENTS.md` "Agentic Setup Maintenance", note in `RELEASE_NOTES.md`.

Each phase leaves the scaffold working (a half-migrated skill still has a valid `SKILL.md`).

## 5. Backward compatibility

- `AGENTS.md.template` Task→Context map references `.ai/skills/om-*/SKILL.md` — those paths **stay valid** (SKILL.md remains, only thinner). New instruction files are additive.
- `STANDALONE.md` removal touches three **verified** references (grep-confirmed): `wizard.ts` `printSummary`, the `shared.ts` conditional copies, and the test **`packages/create-app/src/lib/agentic-skills-standalone-overlays.test.ts`** — which currently *asserts the opposite* (that the 7 skills MUST ship `STANDALONE.md`). That test MUST be inverted/replaced (see §6). (`create-app/AGENTS.md` does **not** reference `STANDALONE.md` — an earlier draft was wrong about this.) No external consumer depends on `STANDALONE.md`.
- **Re-running `agentic:init` on an existing app** must regenerate skill dirs cleanly (overwrite, not merge) so no orphan `STANDALONE.md` survives in a user's `.ai/skills/`.
- These are generated scaffold assets, not a published API surface; no semver break. Record the structure change in `RELEASE_NOTES.md` per house convention.

## 6. Test coverage

- **Conformance guard (new):** for every `agentic/shared/ai/skills/*/SKILL.md` — (a) has frontmatter with non-empty `description`; (b) is within a line budget (e.g. ≤ 60 lines); (c) every reference-map link resolves to an existing file; (d) contains no `## ` procedure-step headings beyond the router sections. Fails CI if a skill regresses to monolithic.
- **No-STANDALONE guard:** assert `find agentic -name STANDALONE.md` is empty. This **replaces** the existing `agentic-skills-standalone-overlays.test.ts` (which enforces the inverse); Phase 1 rewrites that file before the first `STANDALONE.md` deletion, so CI never goes red mid-migration.
- **No-stale-dist assertion:** after `build.mjs`, assert `dist/agentic/**` contains no `STANDALONE.md` (guards the stale-`dist/` regression).
- **Placeholder guard:** assert no copied scaffold file contains a literal `{{PROJECT_NAME}}` after generation.
- **Generator (`shared.test.ts`):** each skill directory is copied recursively (asserts a `workflow/` and a `subagents/` file survive for a sample skill); `.ai/agentic.config.json` is written with the answered `pr.baseBranch`.
- **Wizard (`wizard.test.ts`):** the base-branch question is parsed; `--agents`/headless path yields a valid config (`baseBranch: "auto"` default).
- **Scaffold smoke (`yarn test:create-app`):** generated app contains thin `SKILL.md` + instruction files + `.ai/agentic.config.json`, and no `STANDALONE.md`.

| ID | Risk / failure scenario | Severity | Affected area | Mitigation | Residual |
|----|--------------------------|----------|---------------|------------|----------|
| R1 | **Over-thinning** — a trigger detail removed from `SKILL.md` frontmatter `description` breaks auto-invocation. | High | skill discovery | §2.1 MUST: triggers stay in `description`; conformance guard asserts non-empty description. | A poorly-summarized description could still under-trigger; caught in scaffold smoke + manual review. |
| R2 | **Install-time has no GitHub remote**, so base-branch cannot be auto-detected at scaffold. | Medium | PR skills | `"auto"` sentinel + runtime `gh` fallback (§2.2). | If `gh` is also unavailable at runtime, skill defaults to `main` and logs it. |

| R4 | **Half-migrated set** during the 21-skill pass leaves inconsistent UX (some thin, some fat). | Low | DX | phased order with a working scaffold after each phase; conformance guard flips to enforcing only after Phase 3. | Temporary inconsistency on the feature branch only. |

## 8. Research / precedent

This aligns the standalone skills with the **canonical Claude Code "Agent Skills" format**: a small `SKILL.md` (YAML frontmatter `name`/`description` + lightweight body) with **progressive disclosure** into `references/` loaded on demand. The restructure simply applies that pattern uniformly and adds the `workflow/` + `subagents/` conventions for multi-step and subagent-spawning skills, plus an environment config — the same separation linters/CLIs (e.g. ESLint flat config, Prettier) use to split *rules* from *runtime config*.
