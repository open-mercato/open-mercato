# SPEC-ENT-009: MFA Enrollment Redirect Notice Bar

## TLDR
**Key Points:**
- Show a non-blocking `Notice` bar on `/backend/profile/security/mfa` when the page is opened via middleware redirect with `reason=mfa_enrollment_required`.
- Reuse the existing query-param UX pattern already present in enterprise record locks: read query flag, render one-time UI state, then clear consumed query params using `window.history.replaceState`.
- Keep middleware and redirect URL contracts unchanged.

**Scope:**
- UI-only enhancement in enterprise security MFA list page.
- Notice is shown only for `reason=mfa_enrollment_required`.
- Dismissing notice keeps user on the same page.

**Concerns:**
- Ensure notice does not reappear after dismiss/reload due to stale query params.
- Preserve backward compatibility for all redirect and middleware contract surfaces.

## Overview
Enterprise security middleware currently redirects non-compliant users to:

`/backend/profile/security/mfa?redirect=%2Fbackend&reason=mfa_enrollment_required`

The redirect context is generated by `resolveMfaEnrollmentRedirect()` and already includes optional `overdue=1` when enforcement deadline is exceeded. Today the MFA page ignores these params, so users do not get clear policy context.

This spec adds a contextual notice bar that explains why the redirect occurred and guides users to enroll MFA immediately.

> **Market Reference**: Keycloak and GitLab both enforce MFA through redirect + high-visibility contextual messaging. We adopt the same high-visibility message principle, but use a non-blocking notice to match Open Mercato backend interaction style and existing query-driven UI patterns.

## Problem Statement
- Redirect carries `reason`/`overdue` context, but MFA page currently does not consume it.
- Users can land on MFA page without understanding this is an enforcement requirement.
- The platform already has established URL-driven transient UI patterns (record lock query flags, flash messages), but MFA enrollment redirect UX has a gap.

## Proposed Solution
### Functional behavior
1. On `SecurityMfaPage` mount, read `reason`, `overdue`, and `redirect` from `useSearchParams()`.
2. Show a `Notice` bar only when `reason === 'mfa_enrollment_required'`.
3. If `overdue === '1'`, use stronger warning copy (deadline exceeded).
4. Include dismiss action; dismiss only hides notice and keeps user on current page.
5. Consume query params (`reason`, `overdue`, `redirect`) with `history.replaceState` so notice is one-time.

### Non-functional behavior
- No API changes.
- No middleware contract changes.
- No changes to auth/session flow.
- i18n keys added in `packages/enterprise/src/modules/security/i18n/en.json`.

### Design Decisions
| Decision | Rationale |
|----------|-----------|
| Use `Notice` (non-blocking) instead of modal dialog | Matches requirement and existing backend UX for informational policy state |
| Trigger only for `mfa_enrollment_required` | Keeps behavior deterministic and avoids speculative mapping layer |
| Dismiss keeps user on current page | Avoids navigation surprise and supports deliberate enrollment flow |
| Query-param consumption on page side | Keeps middleware focused on policy routing, UI remains presentation concern |

### Alternatives Considered
| Alternative | Why Rejected |
|-------------|-------------|
| Blocking dialog (`Dialog`) | Rejected by product decision; too disruptive for this flow |
| Reuse `flash/type` URL toast | Flash is too transient and low-emphasis for security enforcement context |
| Handle reason mapping in middleware | Middleware should not own UI semantics; violates separation of concerns |

## User Stories / Use Cases
- A user without enrolled MFA is redirected to MFA page and sees a clear notice that enrollment is required.
- A user past enforcement deadline sees an elevated warning variant of the same notice.
- A user dismisses the notice and remains on the MFA page without forced navigation.

## Architecture
### Relevant existing flow
1. Backend or frontend page middleware executes for authenticated route.
2. `resolveMfaEnrollmentRedirect()` checks policy compliance.
3. If enforced and non-compliant, returns redirect to MFA page with query params.
4. Next.js catch-all redirects user to that URL.

### New UI flow
1. `SecurityMfaPage` reads current search params.
2. `reason === 'mfa_enrollment_required'` enables local `showEnrollmentNotice` state.
3. Page clears consumed params via `replaceState`.
4. Notice renders with standard/due-overdue message.
5. User dismisses notice; only local state changes.

### Implementation notes
- Reuse defensive URL handling pattern from `record_locks` query-flag consumers.
- Keep logic local to MFA page unless extracted as a tiny helper function if testability needs it.

## Risks & Impact Review
### Data Integrity Failures
No write operations introduced. No data integrity risk.

### Cascading Failures & Side Effects
#### Notice fails to render because of client-side parsing error
- **Scenario**: URL parsing or state init fails in MFA page client component.
- **Severity**: Low
- **Affected area**: MFA profile UI only.
- **Mitigation**: Defensive parsing and fallback to hidden notice; keep existing page functional.
- **Residual risk**: User misses enforcement context but can still enroll MFA.

### Tenant & Data Isolation Risks
#### Cross-tenant data exposure through notice content
- **Scenario**: notice renders tenant/user-specific sensitive data from query params.
- **Severity**: Low
- **Affected area**: Security profile UI.
- **Mitigation**: Notice content is static i18n text; do not render raw `redirect` value.
- **Residual risk**: Minimal; no scoped entity data involved.

### Migration & Deployment Risks
#### Contract drift between middleware redirect params and UI parser
- **Scenario**: middleware changes `reason` token in future, UI silently stops showing notice.
- **Severity**: Medium
- **Affected area**: Enforcement UX clarity.
- **Mitigation**: Keep token explicit (`mfa_enrollment_required`) and preserve/extend tests around redirect + UI behavior.
- **Residual risk**: Medium-low; detectable by integration tests.

### Operational Risks
#### Repeated notice display due to missing URL cleanup
- **Scenario**: query params are not removed and notice appears repeatedly on navigation.
- **Severity**: Medium
- **Affected area**: User experience on MFA page.
- **Mitigation**: Explicit `history.replaceState` cleanup after first consumption.
- **Residual risk**: Low.
