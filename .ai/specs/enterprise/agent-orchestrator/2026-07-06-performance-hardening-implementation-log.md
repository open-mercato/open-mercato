### 2026-07-06 — Wave B complete (Phases 2 + 3)
- **Phase 2**: `lib/runtime/admission.ts` (FIFO-among-admissible gate, global + per-tenant caps, bounded wait/queue, idempotent release, `AgentCapacityError.retryable = true`, `isAgentCapacityError`); `agentRuntime.ts` acquires before any DB write for top-level runs only (nested detected via `ctx.parentRunId` OR `getCurrentRunId()`), releases in `finally`, and races `runInProcess` against a cancellable unref'd `OM_AGENT_RUN_TIMEOUT_MS` deadline (failRun exactly once, late model settle swallowed, `AgentRunTimeoutError`); playground route maps capacity → 429 + `Retry-After`, timeout → 422; core `activity-worker-handler.ts` rethrows structurally-retryable errors for queue retry (duck-typed local guard — no enterprise import). i18n `errors.capacity`/`errors.timeout` in en/es/de/pl. - **Observed exhausted-retry semantics** (documented honestly): after the queue's final retry the job ends `failed`/dead-letter and the step-level fail-stop never fires — the instance stays `PAUSED` parked on the proposal-ready signal, resumable by a manual signal. Matches the spec's accepted trade-off; surfaced in the runbook's monitoring checklist (queue depth / failed jobs).
- **Phase 2 deviation**: the playground route's error bodies follow its existing plain-string pattern (values mirror the new `errors.*` locale entries); server-side `resolveTranslations` was not introduced because no error path in that route uses it today.
