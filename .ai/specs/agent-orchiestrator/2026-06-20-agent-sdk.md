## TLDR

The **agent is the primitive**: author one in ~20 lines with `defineAgent` (instructions + tools + skills + model + loop + a typed `result`), auto-discovered from a module's `agents.ts` like `events.ts`. Call it from anywhere as a plain function — `agentRuntime.run(id, input, ctx)` — and get back a typed `AgentResult` that is either **informative** (an answer) or **actionable** (a proposal the caller executes or gates). Built entirely on `ai_assistant`'s `runAiAgentObject` object-mode (so propose-only is *structural* — object-mode passes no tools to the model). A workflow step is **one optional caller**, composed today from existing `EXECUTE_FUNCTION`+`WAIT_FOR_SIGNAL` activities (zero core-workflows change). Everything heavier — dispatch/A2A, identity-OAuth, guardrails beyond schema-validation, eval/metrics, lifecycle, compliance, context-plane, full cockpit — is a deferred overlay.

## Overview

The prior design made *workflow orchestration + regulated decisioning + multi-runtime dispatch* the foundation, so every agent inherited dispatch, identity, compliance, eval, and cockpit machinery before it could run once. This spec inverts that: the callable agent is the base layer, and every heavy concern wraps it only when a use case needs it. The core ships three things — a `defineAgent` authoring surface, an in-process `agentRuntime`, and the `AgentResult` contract — plus a thin persistence layer and one optional workflow composition. Nothing from the overlay specs is deleted; it is re-tiered.

## Problem Statement

Application code needs to author an agent (system prompt + tools + skills + bounded loop + typed output), invoke it synchronously, and act on a typed result — without first standing up dispatch transports, an OAuth identity server, a guardrail engine, an eval harness, or a compliance plane. Today none of that callable primitive exists; only `ai_assistant`'s lower-level `runAiAgentObject` does. The gap is a thin, house-style SDK over it: discovery, a stable agent-id + result-schema contract, attribution, and a minimal run/proposal record — with everything else opt-in.

## Proposed Solution — the 3 things

1. **`defineAgent` + `agents.ts` auto-discovery** — declare an agent (id, instructions, skills, tools, model, loop, `result{kind, schema}`); the generator discovers it exactly like `events.ts`.
2. **`agentRuntime.run(id, input, ctx)`** — resolves the agent, runs `runAiAgentObject` object-mode under the model factory and caller ctx, validates output against `result.schema`, writes a thin `AgentRun`, returns a typed `AgentResult`.
3. **`AgentResult = informative{data} | actionable{proposal}`** — the caller reads `data`, or executes / gates / surfaces the `proposal`; an optional `executeProposal(proposal, ctx)` helper runs proposed actions through OM Commands (audited).

## Architecture — agent as callable primitive

```
agents.ts (defineAgent, as const)            agentRuntime.run(id, input, ctx)
        │  (generator → agents.generated.ts)         │  (in-process; NOT an HTTP endpoint)
        ▼                                            ▼
 AgentDefinition { instructions, skills,   resolve def → object-mode AiAgentDefinition
   tools, model, loop,                       (executionMode:'object', output.schema = result.schema,
   result:{ kind, schema } }                  readOnly, READ-only allowlisted tools)
        │                                            │  attribution: caller ctx + agent-principal
        │                                            ▼
        └──────────────────▶  runAiAgentObject({ executionMode:'object', output:{ schema } })
                                                     │  (object-mode passes NO tools → cannot mutate)
                      validate output vs schema ◀────┤
                                                     ▼
                      write thin AgentRun (Command)  ──▶  AgentResult union returned to caller
                                  │  if actionable
                                  ▼
                      write AgentProposal (Command)  ──▶  caller: executeProposal | gate | UI
```

The runtime is in-process and synchronous-ish (`await`). A **workflow step is one optional caller** — for durable/long-running/human-in-the-loop processes it composes `EXECUTE_FUNCTION → agentRuntime.run → WAIT_FOR_SIGNAL` (see Optional below). The plain service call is the default caller.

## The Agent Definition — `defineAgent`

Authored in a module's `agents.ts` (module root, auto-discovered by `yarn generate`, emitting `agents.generated.ts` — additive surface following the `events.ts` pattern):

```typescript
export const agents = [
  defineAgent({
    id: 'deals.health_check',                 // module.agent — addressable, STABLE key
    instructions: 'Assess a deal’s health and propose a next action.',
    skills: ['sales.deal-playbook'],          // SKILL.md packs (progressive disclosure)
    tools: ['deals.read', 'deals.list_activities'], // defineAiTool ids, allowlisted, READ-only
    model: { /* AiModelFactory resolution: provider/model, tenant precedence */ },
    loop: { maxSteps: 4 },                     // ai_assistant loop controls (stopWhen/budget optional)
    result: { kind: 'actionable', schema: dealNextActionSchema }, // Zod → output.schema
  }),
] as const
```

- **`id`** is `module.agent`, globally unique, and a **STABLE contract** (`BACKWARD_COMPATIBILITY.md`).
- **`result.kind`** is `'actionable' | 'informative'`; **`result.schema`** is a Zod schema in `data/validators.ts`, re-exported from `index.ts`, and *is* the `runAiAgentObject` `output.schema` (single source — a runtime assertion guarantees no drift).
- **`tools`** resolve to existing `defineAiTool`s and are READ-only in MVP (propose-only is structural — see Runtime).
- The MVP deliberately drops the per-capability ACL/context/guardrail/runtime *binding ceremony* from the deeper spec; for the full typed `capabilities.ts` registry (versioned `capability@v`, ACL/guardrail/context facets, dispatch `runtimeRef`) adopt the consolidated spec when needed.

## The Runtime — `agentRuntime`

`AgentRuntimeService` (DI key, `lib/orchestration/`) thinly wraps `ai_assistant`'s `runAiAgentObject` object-mode. `run(id, input, ctx)` (ctx carries `tenantId`/`organizationId`/`userId`):

1. **Resolve** `id` → `AgentDefinition` → an object-mode `AiAgentDefinition` (`executionMode:'object'`, `output.schema = result.schema`, `readOnly:true`, `mutationPolicy:'read-only'`, READ-only `allowedTools`); load `skills` (SKILL.md packs).
2. **Attribute.** Run under the caller `ctx`, scoped org/tenant, tagged with lightweight **agent-principal attribution** (so the run/proposal record carries "produced by agent `id` on behalf of `userId`"). No OAuth identity server — that pairs with external runtimes and is deferred.
3. **Run.** `runAiAgentObject({ executionMode:'object', output:{ schema } })` under `AiModelFactory` (provider/model/tenant precedence) with `loop` controls. Object-mode passes **no tools** to `generateObject`/`streamObject` (verified: `agent-runtime.ts` `void tools`) — a single typed structured-output call, so the model cannot mutate.
4. **Validate** output against `result.schema`; an invalid output is recorded as a failed run, never a malformed result.
5. **Persist** a thin `AgentRun` via the **Command path** (audit/events/index fire); if `result.kind === 'actionable'`, also persist an `AgentProposal`. Attribution makes these the agent's only "writes", through the audited path.
6. **Return** the typed `AgentResult` union.

No dispatch, queue, or A2A on the common path — the only ingress is calling the DI service. (External/firewalled runtimes are a deferred overlay behind an identical `AgentResult` contract.)

## The Result Contract

```typescript
type AgentResult<T> =
  | { kind: 'informative'; data: T }                       // an answer; caller reads it
  | { kind: 'actionable'; proposal: {
        actions: ProposedAction[];   // typed, ready to execute
        confidence?: number;
        rationale?: string;
      } }
```

- **Informative** → the caller uses `data` (summary, classification, extraction, answer).
- **Actionable** → the agent **proposes**; execution stays with the caller: run immediately, hand to a gate (auto-approve vs human task), or surface in UI. This preserves "LLM proposes, the app disposes" **without** mandating a workflow or rules engine.
- **`executeProposal(proposal, ctx)`** (optional helper) runs the proposed actions through OM Commands so the effects are audited. Callers may instead execute or gate the proposal themselves.

## Workflow Step (`INVOKE_AGENT`) — MVP-core for the hackathon

Architecturally a workflow step is *one optional caller* of `runAgent`; **for the hackathon cut it is core** (the new-user headline). For durable / multi-actor / human-in-the-loop processes the **same agent** runs as a workflow step. The semantics:

```
workflow ──▶ [EXECUTE_FUNCTION: agentRuntime.run(id, input)] ──▶ writes AgentProposal
          ──▶ [WAIT_FOR_SIGNAL] parks until disposition
          ──▶ [disposition] confidence ≥ threshold → auto-approve ──▶ resume signal: agent.proposal.ready
                            else raise USER_TASK (My Tasks card)   ──▶ human Approve/Edit/Reject ──▶ resume
          ──▶ [EXECUTE_FUNCTION: effector] runs the approved action (OM Command, audited)
```

MVP disposition is a **single threshold** (`confidence ≥ x → auto-approve`, else `USER_TASK`) — a plain check or one `business_rules` VALIDATION rule. Informative results just store `data` and proceed.

> **Approved (2026-06-20):** ships as a **first-class `INVOKE_AGENT` activity** in core `workflows` — the `workflows` owner (Patryk Lewczuk) approved the additive change, so the diagram above is promoted from a two-activity composition to one editor node + one executor case (same park/resume semantics). ## Data Models

Lean and additive. Both entities follow the normative conventions (MikroORM **v7** `/legacy`, explicit `@Property`, dual `tenant_id`+`organization_id`, soft-delete, `updated_at` for editable rows, `agent_` table prefix, FK ids only — no cross-module ORM relations). **Explicitly NO token/cost/eval/span columns in MVP.**

```typescript
import { OptionalProps } from '@mikro-orm/core'
import { Entity, Index, PrimaryKey, Property } from '@mikro-orm/decorators/legacy'

export type AgentRunStatus = 'running' | 'completed' | 'failed'

@Entity({ tableName: 'agent_runs' })
@Index({ name: 'agent_runs_tenant_org_idx', properties: ['tenantId', 'organizationId'] })
@Index({ name: 'agent_runs_agent_idx', properties: ['organizationId', 'agentId'] })
export class AgentRun {
  [OptionalProps]?: 'status' | 'createdAt'

  @PrimaryKey({ type: 'uuid', defaultRaw: 'gen_random_uuid()' })
  id!: string

  @Property({ name: 'tenant_id', type: 'uuid' })
  tenantId!: string

  @Property({ name: 'organization_id', type: 'uuid' })
  organizationId!: string

  @Property({ name: 'agent_id', type: 'varchar', length: 100 })
  agentId!: string // defineAgent id; NOT an ORM relation

  @Property({ name: 'input_ref', type: 'jsonb', nullable: true })
  inputRef?: any | null // input payload or a reference to it

  @Property({ name: 'output', type: 'jsonb', nullable: true })
  output?: any | null // validated AgentResult; shape enforced by result.schema Zod

  @Property({ name: 'status', type: 'varchar', length: 20, default: 'running' })
  status: AgentRunStatus = 'running'

  @Property({ name: 'created_at', type: Date, onCreate: () => new Date() })
  createdAt: Date = new Date()
}
```

```typescript
export type AgentProposalDisposition =
  | 'pending' | 'auto_approved' | 'approved' | 'edited' | 'rejected'

@Entity({ tableName: 'agent_proposals' })
@Index({ name: 'agent_proposals_tenant_org_idx', properties: ['tenantId', 'organizationId'] })
@Index({ name: 'agent_proposals_run_idx', properties: ['organizationId', 'runId'] })
export class AgentProposal {
  [OptionalProps]?: 'disposition' | 'createdAt' | 'updatedAt' | 'deletedAt'

  @PrimaryKey({ type: 'uuid', defaultRaw: 'gen_random_uuid()' })
  id!: string

  @Property({ name: 'tenant_id', type: 'uuid' })
  tenantId!: string

  @Property({ name: 'organization_id', type: 'uuid' })
  organizationId!: string

  @Property({ name: 'agent_id', type: 'varchar', length: 100 })
  agentId!: string

  @Property({ name: 'run_id', type: 'uuid' })
  runId!: string // FK id → agent_runs

  @Property({ name: 'payload', type: 'jsonb' })
  payload!: any // { actions, confidence?, rationale? }; shape enforced by result.schema Zod

  @Property({ name: 'confidence', type: 'float', nullable: true })
  confidence?: number | null

  @Property({ name: 'disposition', type: 'varchar', length: 20, default: 'pending' })
  disposition: AgentProposalDisposition = 'pending'

  @Property({ name: 'created_at', type: Date, onCreate: () => new Date() })
  createdAt: Date = new Date()

  @Property({ name: 'updated_at', type: Date, onCreate: () => new Date(), onUpdate: () => new Date() })
  updatedAt: Date = new Date() // optimistic-lock header source for dispose

  @Property({ name: 'deleted_at', type: Date, nullable: true })
  deletedAt?: Date | null
}
```

## API Contracts

The runtime is **in-process, not an endpoint** — the only ingress is calling `agentRuntime.run(...)`. HTTP surfaces are read + the dispose write:

- **Read (`makeCrudRoute` + `indexer` + `export const openApi`):**
  - `GET /api/agent_orchestrator/agents` — list discovered agents (id, result kind, tools, skills).
  - `GET /api/agent_orchestrator/runs/:id` — one run (input ref, output, status); list/detail org-scoped, returns `updatedAt` where applicable.
- **Write (non-CRUD, Command + mutation-guard + optimistic-lock):**
  - `POST /api/agent_orchestrator/proposals/:id/dispose` — body `{ disposition, editedPayload?, reason? }`. Runs through the Command pattern (`validateCrudMutationGuard` before, `runCrudMutationGuardAfterSuccess` after), enforces optimistic locking (`enforceCommandOptimisticLock`) on the proposal's `updatedAt`, surfaces 409s via `surfaceRecordConflict`; an approve disposition triggers the effector / resume signal.
- **Events** (`events.ts`, `createModuleEvents`, `as const`, `module.entity.action` past tense): `agent_orchestrator.run.completed`, `agent_orchestrator.run.failed`, `agent_orchestrator.proposal.ready`.
- **ACL** (`acl.ts` + `setup.ts` `defaultRoleFeatures`): `agent_orchestrator.invoke`, `agent_orchestrator.view`, `agent_orchestrator.proposal.dispose`.

## Phases

1. **Core SDK.** `defineAgent` + `agents.ts` auto-discovery (generator → `agents.generated.ts`); `agentRuntime.run` over `runAiAgentObject` object-mode; `AgentResult` union; output-schema validation; agent-principal attribution. *(P0)*
2. **Thin persistence + dispose.** `AgentRun` + `AgentProposal` entities + migration/snapshot; read routes (`/agents`, `/runs/:id`) via `makeCrudRoute`+`openApi`; `POST /proposals/:id/dispose` via Command + mutation-guard + optimistic-lock; events; ACL; optional `executeProposal` helper. *(P0)*
3. **Workflow `INVOKE_AGENT` step (core for hackathon).** First-class additive `INVOKE_AGENT` activity in core `workflows` (owner-approved) + the one editor node (agent · input · auto-approve threshold); parks → `agentRuntime.run` → single-threshold disposition (auto-approve else `USER_TASK`) → resume on `agent_orchestrator.proposal.ready`; one effector step. *(P0 — headline)*

## Acceptance

- An agent declared once in `agents.ts` produces a typed `id` and is invocable via `agentRuntime.run` with no second declaration of its id or result schema.
- `agentRuntime.run('x.y', input, ctx)` runs an object-mode LLM under the caller ctx, validates output against `result.schema`, persists a thin `AgentRun`, and returns the typed `AgentResult` union.
- An `actionable` result persists an `AgentProposal`; the agent holds NO mutating tool — its only writes are `AgentRun`/`AgentProposal` via the audited Command path.
- An invalid model output is recorded as `run.failed`, never a malformed result.
- `POST /proposals/:id/dispose` enforces RBAC, tenant isolation, and optimistic locking (409 on stale `updatedAt`).
- (Optional) A workflow step runs the agent, parks on `WAIT_FOR_SIGNAL`, and resumes on `agent_orchestrator.proposal.ready` after disposition.

## Risks & Impact Review

| Scenario | Severity | Area | Mitigation | Residual |
|---|---|---|---|---|
| Future change passes mutating tools to object-mode, breaking structural propose-only | High | security | `readOnly:true` + `mutationPolicy:'read-only'`; READ-only allowlist; release-gate test asserts no `isMutation:true` tool reachable | Low |
| `result.schema` drifts from `agent.output.schema` | Medium | correctness | Single source in `data/validators.ts`; runtime asserts `result.schema === output.schema` | Low |
| Cross-tenant run/proposal read | High | tenancy | Dual `tenant_id`+`organization_id`, org-scoped reads, cross-tenant denial test | Low |
| Concurrent dispose causes lost update | Medium | data-integrity | `updated_at` + `enforceCommandOptimisticLock`; 409 surfaced via `surfaceRecordConflict` | Low |
| Agent id / result schema treated as ad-hoc, breaking callers | Medium | contract | Ids + result schemas are STABLE; additive evolution only; generator surface follows `events.ts` | Low |
| Generator change is a new auto-discovery contract surface | Low | build | Additive `agents.ts` file type mirroring `events.ts`; no existing surface changes | Low |
| Pre-supplied context under-serves retrieval-heavy agents | Medium | answer quality | Caller-supplied context or a `retrieve` tool now; full context-plane is a deferred overlay | Medium |

## Migration & Backward Compatibility

- **Agent ids and result schemas are STABLE contracts** (`BACKWARD_COMPATIBILITY.md`). A published id is frozen; an incompatible result-schema change ships under a new agent id (or a versioned overlay) with the old retained ≥1 minor per the deprecation protocol.
- **Everything here is additive.** `agents.ts` / `agents.generated.ts` is a new auto-discovery surface following the `events.ts` pattern; `agent_runs` / `agent_proposals` are net-new tables; events, ACL features, and the read/dispose routes introduce no existing-id changes.
- The deeper consolidated spec layers a *versioned* `capabilities.ts` registry on top; `agent_id` here maps forward to a capability `key@v` without breaking stored runs.
- Run `yarn generate && yarn db:generate` (review SQL + snapshot) → `yarn typecheck && yarn lint && yarn test`.
