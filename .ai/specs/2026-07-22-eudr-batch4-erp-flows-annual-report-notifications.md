# EUDR Compliance Batch 4 — ERP-Native Flows, Annual Report, Supplier Panel & Notifications

## TLDR

Batch 4 of the `eudr` core module (batches 1–3 on the same unmerged branch: mappings/evidence/DDS + plots/risk/lifecycle + pickers/cockpit/order-panel/search). Competitive research 2026-07-22 (LiveEO TradeAware, Coolset, Odoo/SprintIT, TrusTrace, Duveka + regulatory sources) shows the remaining in-platform gaps are ERP-native flow depth and engagement surfaces — not new compliance math. Five slices:

1. **Order→DDS prefill + statement duplication** — the order-compliance panel's "Create statement" carries the order context; the statement create page seeds from `?orderId=` (order picker + commodity suggestion from mapped order lines + title template) and from `?duplicateFrom=` (repeat-shipment duplication, the Coolset "repeat low-risk shipment" mechanic without automation risk). Odoo/Duveka's core wedge: DDS creation starts inside the sales flow.
2. **Wood species names** — `species_scientific_name`/`species_common_name` on product mappings. Scientific + common name are mandatory DDS content for timber (Art. 9(1)(a); every TRACES guide lists them); no current field captures them. Export packet includes them; export readiness and the statement detail UI gain a non-blocking warning when either name is missing on a wood mapping.
3. **Art. 12(3) annual due-diligence report** — aggregate JSON/CSV export per calendar year (statements, commodities, quantities, countries with risk tiers, risk outcomes, mitigation summary — aggregates only, zero PII), computed in SQL, per-block feature-gated like the cockpit. Non-SME operators must publish annually (first report due after 2027-12-30); competitor tooling for it is still "emerging" — cheap differentiator on data we already hold.
4. **Supplier EUDR readiness panel** — injection widget registered on BOTH company detail generations: `detail:customers.company:footer` (companies-v2, the page primary navigation actually routes to) and `customers.company.detail:details` (legacy v1 page), fed by a new per-supplier aggregate route. The Duveka "supplier responsibility" surface without a portal.
5. **Notifications** — org-safe in-app notifications for lifecycle moments (statement submitted / reference issued / withdrawn, non-negligible risk conclusion, mitigation completed) via new lifecycle events emitted by the commands that already detect the transitions, `notifications.ts` types, idempotent persistent subscribers (deterministic `groupKey`), and feature-based fan-out restricted to the event's organization.

Plus **polish fixes** from the batch-4 audit: risk-assessment custom-entity `labelField` currently `statementId` (a raw UUID — violates the name-based-UX mandate) → `conclusion`; cockpit rollups per-feature gated like the queues (closes a cross-feature aggregate leak); OpenAPI drift (submissions `plotIds`, plots `geometry` presence note).

Out of scope (roadmap, unchanged): TRACES SOAP/API live submission and polling, supplier self-service portal, satellite deforestation screening, volume reconciliation, lot/serial binding, leaflet-draw editing, AI document extraction. Known limitation: DataTable list-filter supplier dropdowns (plots/submissions lists) show the first 100 companies pending a FilterBar async-typeahead primitive — every FORM picker is a server-searchable typeahead.

## Resolved assumptions (autonomous defaults)

- **Q1 One spec or five?** One batch spec, phases = capabilities — the shipped house precedent for this module (batches 1–3); each phase is independently revertable and the branch is unreleased. *(Most reversible: keeps the existing delivery model.)*
- **Q2 Supplier panel host spot?** Dual registration. Primary: `detail:customers.company:footer` on companies-v2 (`companies-v2/[id]/page.tsx:609`) — the page every primary navigation path targets (companies list name link `companies/page.tsx:617`, row click `:890`, row action `:910`); its `injectionContext` carries `companyId`/`resourceId` (`:175`). Legacy: `customers.company.detail:details` (v1 `companies/[id]/page.tsx:970`) so direct links to the old route keep the panel; the widget normalizes the two context shapes to obtain the company id. *(Spec-jury F-04.)*
- **Q3 Notification recipients?** Feature-based fan-out restricted to the event's organization: the notifications service `createForFeature` currently resolves recipients tenant-wide (`getRecipientUserIdsForFeature(db, ctx.tenantId, feature)`), which would write notification rows for users with no access to the source organization. Batch 4 makes ONE additive released-module change (documented like batch 2's attachments touch): `createForFeature` input gains optional **boolean** `restrictRecipientsToOrganization?: boolean`; when true, the organization is taken from `ctx.organizationId` — the SAME value stamped on the created rows, so the recipient filter and the row scope can never diverge (no separate org parameter exists). Candidates are filtered through the platform RBAC service (`userHasAllFeatures(candidateId, [requiredFeature], { tenantId, organizationId: ctx.organizationId })`, resolved lazily from the container), with a hard candidate cap of 200 (beyond it the fan-out is skipped + logged — a tenant with >200 holders of a manage feature needs a digest design, not 200 rows per transition). If the RBAC filter cannot run or `ctx.organizationId` is null while the flag is set, the fan-out is skipped entirely and logged (deliberate fail-closed exception to the batch's fail-open default: better no notification than a cross-org one). The eudr helper always sets the flag and passes the event's org as `ctx.organizationId`. **The write-side filter is the isolation mechanism** — the notifications read path is NOT relied upon for org isolation (rows are stamped with the source org, but general list reads are not uniformly org-filtered), which is exactly why recipients must be authorized before a row is written. Recipient predicate semantics (per the platform RBAC): `userHasAllFeatures(candidate, [feature], { tenantId, organizationId })` returns true for users whose ACL grants the feature **in that organization context** — org-restricted roles are denied for other orgs, while tenant-wide unrestricted roles pass for every org; a tenant-wide admin legitimately receives cross-org compliance notifications (they can open the record), and users org-restricted elsewhere receive nothing. Statement lifecycle → `eudr.statements.manage` holders; risk/mitigation → `eudr.risk.manage` holders. *(Spec-jury round-1 blocker F-01; round-2: flag bound to ctx; round-3: false read-path-filtering claim corrected, RBAC predicate semantics stated.)*
- **Q4 Species fields on mapping or submission?** Product mapping — species is a product attribute (Odoo stores it on the product master); submissions inherit it through their mapping. Nullable, wood-relevant, warning-level only. *(Smallest schema surface.)*
- **Q5 Annual report shape?** `GET /api/eudr/reports/annual?year=YYYY` (+`format=csv`), year bucketed by `submitted_at` (UTC), aggregates computed in SQL, per-block feature gating (see API contract), zero PII. Surfaced as a cockpit card (year select + totals + download buttons). *(Read-only, no schema change.)*
- **Q6 What does Duplicate copy?** Structural fields only: title (via `eudr.statements.duplicateTitle` template), commodity, activityType, actorRole, orderId (the order picker re-captures the order snapshot when it resolves the seeded id — `formConfig.tsx` fires `onSnapshot` on resolution), quantityKg, supplementaryUnit/Quantity, notes. Never: status (forced draft), reference/verification numbers, referencedStatements, submittedAt/referenceIssuedAt, linked submissions (evidence is per-consignment). *(No write-path change: duplication is create-page seeding.)*
- **Q7 Which tests does the cockpit rollup gating change touch?** The unit gating suite `api/__tests__/compliance-overview-gating.test.ts` is extended for the newly gated rollup blocks; TC-EUDR-008 gains new partial-feature assertions (its existing full-feature admin calls keep passing — wildcard grants see every block). *(Corrected per spec-jury F-32.)*
- **Q8 `labelField` replacement for risk assessments?** `conclusion` — short, meaningful enum. Non-unique labels are the accepted precedent for EAV label surfaces (the mapping's `labelField: 'commodity'` is equally non-unique); `statementId` is a raw UUID and `assessed_by_name` may hold an email. *(Spec-jury nit F-30 acknowledged.)*

## Problem Statement

A sales operator who sees the EUDR panel on an order must leave the order, open Compliance → Statements → New, and re-find the same order by hand; repeat consignments are re-typed from scratch. Timber statements cannot record the legally required species names anywhere. The Art. 12(3) annual public report — a legal duty for non-SME operators from FY2027 — has no aggregation surface. A company (supplier) page says nothing about that supplier's EUDR posture. Nothing notifies compliance staff when a statement's reference is issued (the moment downstream partners wait for), when a risk conclusion comes back non-negligible, or when mitigation completes. And the batch-4 audit found one raw-UUID label leak (`ce.ts` risk-assessment `labelField`) plus a cross-feature aggregate leak in the cockpit rollups.

## Proposed Solution

Extend the `eudr` module in place — same architecture (zod validators, undoable commands, `makeCrudRoute` + hand-written scoped routes, DataTable/CrudForm, injection widgets, i18n ×4). All changes additive. Exactly one file outside `eudr/` changes: the notifications service gains the optional org-restricted fan-out input (additive, BC-permitted event of the same class as batch 2's attachments payload addition).

### Design Decisions

| Decision | Rationale |
|----------|-----------|
| **Create-page seeding via query params** (`?orderId=`, `?duplicateFrom=`), not a new API | The risk-assessments create page already prefills from `?statementId=` — shipped house mechanism. Seeding fetches the source record through existing list routes' `?id=` detail reads; nothing new server-side. `useSearchParams()` is allowed for query strings (the `useParams()` ban covers route segments only — repo lesson 2026-07-06). **Precedence**: `duplicateFrom` wins when both params are present; `orderId` is then ignored and an info flash says so. Invalid, foreign-org, or missing source ids → empty form + one `eudr.statements.prefillUnavailable` info flash; never an error page. |
| **Commodity suggestion resolves order lines through the real order-lines route** | Two-step: `GET /api/sales/orders?id=<id>` for the summary/snapshot (`orderNumber`), then `GET /api/sales/order-lines?orderId=<id>&pageSize=100` paged until complete with a 3-page cap (>300 lines → treat as ambiguous). Distinct productIds → `GET /api/eudr/product-mappings?productId=<csv>`; exactly one distinct in-scope commodity → preselect, else leave empty. Any partial/failed fetch leaves commodity empty (no flash for the commodity alone — it is best-effort). Quantity is **not** auto-filled (order line units ≠ net mass kg; a wrong prefilled quantity is worse than an empty field). Sales module absent → seeding degrades to an empty form (existing picker soft-degradation). *(Spec-jury F-05.)* |
| **Duplicate = seeded create, not a server-side clone command** | Forced `status='draft'` create path reuses every existing guard (batch-2 "create restricted to draft"); no new mutation surface, per-field copy list fixed in the page (Q6). |
| **Species on `EudrProductMapping`, warning-level enforcement, both names checked** | Two nullable text columns + additive migration. Mapping form shows the fields only when `commodity === 'wood'` (CrudForm native `visibleWhen: { field: 'commodity', equals: 'wood' }`). Export packet `productMappings[]` gains both fields; export `readiness` gains an additive `warnings: string[]` including `eudr.warnings.speciesMissing` when a linked **wood** mapping lacks EITHER trimmed name (scientific or common — both are mandatory DDS content). The same warning renders as an advisory chip in the statement detail readiness/lifecycle area so it is visible before export. The submit gate is NOT extended — species is DDS content, not a lawfulness precondition; blocking would contradict the no-blockers mandate (residual documented in Risks). *(Spec-jury F-06, F-34.)* |
| **Annual report: SQL aggregation, per-block ACL, safe CSV** | Hand-written route; `resolveOrganizationScopeForRequest` + tenant filter (batch-1 export precedent). All aggregates are computed **in SQL** (kysely `GROUP BY` over statements / submissions / assessments / mitigation actions) — no row materialization, no unbounded scans regardless of tenant size; **every query filters `deleted_at IS NULL`** on its table, and raw SQL numerics (COUNT/SUM strings or bigints from the driver) are normalized deterministically before zod validation (counts → number, quantity sums → 3-decimal strings via the lib accumulator). The additive migration adds a supporting partial index `eudr_due_diligence_statements (tenant_id, organization_id, submitted_at) WHERE deleted_at IS NULL`. Per-block feature gating mirrors the cockpit: `statements` block (and the CSV, which contains only statement-derived rows) → base `eudr.statements.view`; `countries` (derived from submissions) additionally requires `eudr.submissions.view`; `risk` + `mitigation` additionally require `eudr.risk.view`; unauthorized blocks are omitted server-side and optional in the zod response schema. CSV serialization REUSES the exported shared serializer `serializeExport(prepared, 'csv')` from `packages/shared/src/lib/crud/exporters.ts` — it already performs CSV quoting and spreadsheet-formula neutralization (`neutralizeSpreadsheetFormula`) on every string cell; `lib/annual-report.ts` only shapes the `PreparedExport` columns/rows and writes no escaping code of its own. `supplementaryUnit` is free text and must never reach a spreadsheet interpretable; a malicious-unit test proves the neutralization end-to-end through the shared serializer. `risk.simplified` is defined as the count of bucketed statements whose LATEST assessment has `is_simplified = true`. *(Spec-jury round-1 blockers F-02, F-03; round-2: soft-delete exclusion, numeric normalization, supporting index; round-3: reuse the exported `serializeExport` instead of local escaping, `simplified` semantics pinned.)* |
| **Annual aggregation semantics (fixed, testable)** | Statement bucket: `submitted_at` ∈ calendar year (UTC) and status ∈ submitted/available/withdrawn/archived — drafts never appear; withdrawn/archived remain in the report with their `byStatus` breakdown (the report reflects what was filed during the year). `byCommodity[]`: count + `quantityKg` (numeric sum) + `supplementaryQuantities: [{ unit, quantity }]` grouped by trimmed, case-folded unit (rendered uppercased — "kg" and "KG" are one bucket), empty/null units omitted, quantities accumulated as integer thousandths and rendered with 3 decimals (no float drift, no cross-unit sums). `countries[]`: distinct origin countries of submissions linked to bucketed statements, each with its benchmarking tier and submission count. `risk`: the LATEST assessment per bucketed statement — counts of negligible / non-negligible / simplified. `mitigation`: actions belonging to those latest assessments — total + completed. *(Spec-jury F-08, F-17, F-24, F-29.)* |
| **Supplier aggregate route feeds the panel in one request** | `GET /api/eudr/suppliers/compliance?supplierEntityId=` → `{ submissions: {total, byStatus, avgCompleteness}, lastSubmissionAt, plots?: {total, withWarnings} }`, base ACL `eudr.submissions.view`; the `plots` block is included only when the caller also holds `eudr.plots.view` (cockpit per-feature omission pattern). The widget hides itself entirely (renders null) when totals are zero or the fetch fails — failures are recorded via the structured logging facade (`createLogger('eudr')`, debug level), never `console.*` and never a visible error on the company page. *(Spec-jury F-31.)* |
| **Lifecycle events named per module convention, emitted by commands** | New event ids use the module's existing entity tokens and past-tense actions: `eudr.due_diligence_statement.submitted`, `eudr.due_diligence_statement.reference_issued` (the submitted→available transition — "reference issued" is the past-tense domain fact; "available" is a state, not an action), `eudr.due_diligence_statement.withdrawn`, `eudr.risk_assessment.concluded`, `eudr.mitigation_action.completed` (category `lifecycle`, `as const`, generator-discovered). Statements: emitted from `commands/statements.ts` after the successful write at the three transitions. Risk: `concluded` emitted on create (always) and on update whenever `conclusion` changed — the subscriber, not the event, decides whether to notify. Mitigation: `completed` emitted both on the update transition that auto-sets `completed_at` AND on create-with-status-completed. Payloads carry `{ id, tenantId, organizationId, title/statementTitle, referenceNumber?, conclusion?, occurredAt }` where `occurredAt` is stamped once at emission (ISO string) — the idempotency anchor. *(Spec-jury F-07, F-23, F-26.)* |
| **Notifications: idempotent subscribers, org-restricted fan-out, correct failure split** | `notifications.ts` declares 5 types with severity, i18n title/body keys, linkHref, and explicit retention: `expiresAfterHours: 168` for submitted/withdrawn/concluded/completed, `720` for reference_issued (the compliance-critical one). Each subscriber (persistent, one side effect) builds the notification with a deterministic `groupKey = <eventId>:<entityId>:<occurredAt>` — the service's `createOrRefreshNotification` dedupes **active** notifications per (recipient, type, groupKey) under an advisory lock, so at-least-once event delivery cannot double-notify the same occurrence while later distinct occurrences (new `occurredAt`) still notify. Known residual (documented, accepted): groupKey coalescing checks active statuses only — a redelivery that arrives after the recipient already read/dismissed the notification re-creates it; the window is the persistent bus's near-term retry horizon, and this is still strictly stronger than the platform baseline (existing modules' notification subscribers have no occurrence dedupe at all). Payload-carried tenant/org scope is the platform's subscriber contract (events originate from our own commands on the internal bus — precedent: attachments completeness subscribers, customers deal-closure notifications); subscribers do not re-load entities. Failure split: notifications MODULE absent / service unresolvable → no-op with a structured log (intentional degradation); service resolved but create THROWS → rethrow so the persistent bus retries (groupKey makes the retry safe). Subscriber payload missing required scope → no-op log. *(Spec-jury round-1 F-01, F-09, F-13, F-15, F-19, F-28, F-33; round-2: coalescing residual + payload-scope precedent documented.)* |
| **Cockpit rollups gated like the queues** | `mappingsInScope` → `eudr.mappings.view`; `submissions.*` → `eudr.submissions.view`; `riskReviewsDueSoon` → `eudr.risk.view`; plots KPI (already queue-gated) unchanged; `statements.*` + `deadline` stay on the base gate. Response omits absent blocks; cockpit page + dashboard widget render only present blocks (they already handle absent queues). Closes the audit's cross-feature aggregate leak. |
| **`ce.ts` risk-assessment `labelField: 'conclusion'`** | Any EAV/label surface would otherwise print a raw UUID — direct violation of the name-based mandate (audit finding). Enum-label precedent: mapping's `commodity`. |

### Alternatives Considered

| Alternative | Why Rejected |
|-------------|--------------|
| Server-side `duplicate` command/route | New mutation surface + undo semantics for something a seeded create does with zero server change. |
| Auto-create DDS when an order contains in-scope products (full Odoo parity) | Write-side coupling from sales flow into eudr (interceptor or subscriber creating records) — over-automation with tenant-visible side effects; prefill keeps the human in charge. Roadmap candidate once TRACES filing lands. |
| Species fields on submissions (per-consignment species) | Species is a property of the traded product, not the consignment; mapping-level keeps one source of truth and zero submission-form churn. |
| Annual report as a background job + stored artifact | Aggregation is SQL-side and bounded; on-demand compute avoids storage, staleness, and a worker. |
| Supplier panel reading existing list routes client-side | 3 fan-out requests per company page view + no single feature gate + duplicated aggregate logic. |
| Tenant-wide notification fan-out (unmodified `createForFeature`) | Writes notification rows (with statement titles) for users without access to the source organization — cross-org data at rest; rejected by the spec jury (F-01). The additive org-restricted filter keeps the service's contract intact for existing callers. |
| A dedicated `eudr.risk_assessment.became_non_negligible` event | The generic `concluded` event (payload carries the conclusion) covers create + update flips with one id and lets future subscribers react to negligible conclusions too; the notification policy lives in the subscriber. |
| Time-based reminder notifications (review due, window expiring) | No cron/repeatable-job primitive in `@open-mercato/queue` today; the cockpit queues already surface both states on read. Event-driven set only; time-based joins the roadmap with the scheduling primitive. |
| Local verification badge for referenced upstream statements | Upstream refs are usually external suppliers' DDS — a local match is the rare intra-org case; batch 2 already hard-blocks withdrawing a downstream-referenced statement. Roadmap nice-to-have. |

## User Stories

- A sales operator on an order clicks "Create statement" and lands in a form where the order is already selected, the commodity already suggested from the order's mapped lines, and the title pre-filled — no re-finding, no typing.
- A compliance officer duplicates last month's statement for the next consignment of the same product and only updates quantity and evidence links.
- A timber compliance officer records *Quercus robur* / "European oak" once on the product mapping; every export packet carries both names; the statement detail and export readiness warn while either is missing.
- A compliance lead downloads the FY2026 Art. 12 report (JSON for the auditor, CSV for the website team) from the cockpit — seeing only the blocks their role may see.
- An account manager opens a supplier company page (v2 or the legacy route) and sees its EUDR posture: 12 plots (2 with warnings), 8 submissions (avg 91% complete), last submission 3 days ago — with links into the filtered lists.
- A compliance manager gets an in-app notification the moment a statement's reference is issued, when a risk assessment concludes non-negligible (on creation or a later edit), and when a mitigation action completes — exactly once per occurrence, only within the right organization.

**Released-module touch (the only file outside `eudr/` + guard/test surfaces):** `packages/core/src/modules/notifications/lib/notificationService.ts` — `createForFeature` input gains optional boolean `restrictRecipientsToOrganization` (additive; existing callers unaffected; the org is always `ctx.organizationId`, never a separate parameter). Implementation fits inside this one file: the resolve path already constructs the service with the DI container (`resolveNotificationService` → `createNotificationService({ em, eventBus, commandBus, container })`, notificationService.ts:686), so the RBAC service is resolved lazily from `deps.container` at call time; a flagged call on an instance built without a container (e.g. the module's `di.ts` wiring) fails closed (skip + log) — and the eudr helper always goes through `resolveNotificationService`. When set, tenant candidates (capped at 200) are filtered per the RBAC predicate in Q3; resolution failure or null ctx org → skip fan-out + structured log (fail-closed). A unit test in the notifications module covers filter-on, filter-off, cap-exceeded, no-container, and rbac-unavailable paths. This is the batch's one deliberate cross-module code change — the sanctioned pattern for a platform-service capability gap, precedented by batch 2's additive attachments-payload and catalog-enricher-opt-in touches (a module-local workaround would require duplicating the notifications module's private candidate resolution, a worse coupling).

No new ACL features (existing view/manage cover every new surface — reads gated by `.view`, no new mutations). No DI changes. `yarn generate` picks up notifications.ts, events, subscribers, widget, routes.

## API Contracts (all additive; org/tenant scoping; OpenAPI exported)

- `GET /api/eudr/reports/annual?year=YYYY[&format=csv]` — base ACL `eudr.statements.view`; `year` validated (2020..2100). JSON: `{ year, generatedAt, statements: { total, byStatus, byCommodity: [{ commodity, count, quantityKg, supplementaryQuantities: [{unit, quantity}] }] }, countries?: [{country, tier, submissionCount}], risk?: { assessments, negligible, nonNegligible, simplified }, mitigation?: { total, completed } }` — `countries` requires `eudr.submissions.view`; `risk`/`mitigation` require `eudr.risk.view`; unauthorized blocks omitted (optional in schema). Aggregates only — no names, nothing encrypted. `format=csv` → statement-derived per-commodity rows only (base ACL sufficient), `text/csv; charset=utf-8`, download disposition, serialized via the shared `serializeExport` (quoting + formula neutralization built in).
- `GET /api/eudr/suppliers/compliance?supplierEntityId=<uuid>` — ACL `eudr.submissions.view`; response `{ submissions: { total, byStatus, avgCompleteness }, lastSubmissionAt, plots?: { total, withWarnings } }` with `plots` omitted unless caller holds `eudr.plots.view`. 400 on missing/invalid id; scoped to tenant+org.
- `GET /api/eudr/dashboard/widgets/compliance-overview` — rollup blocks become per-feature conditional (see Design Decisions); zod responseSchema updated (all gated blocks optional); queues unchanged.
- `GET /api/eudr/statements/[id]/export` — `productMappings[]` items gain `speciesScientificName`/`speciesCommonName`; `readiness` gains additive `warnings: string[]` (i18n keys; `eudr.warnings.speciesMissing` when a linked wood mapping lacks either trimmed name).
- `/api/eudr/product-mappings` — create/update schemas + item schema gain the two species fields (optional, trimmed, ≤256).

## Commands & Events

- New event declarations (`events.ts`, `as const`, category `lifecycle`): `eudr.due_diligence_statement.submitted`, `eudr.due_diligence_statement.reference_issued`, `eudr.due_diligence_statement.withdrawn`, `eudr.risk_assessment.concluded`, `eudr.mitigation_action.completed`. ## Edge Cases & Failure Scenarios

- Seeding source unreadable (deleted statement, foreign org order, sales module absent) → empty form + info flash; both params → duplicateFrom wins; values that are not uuid-shaped are ignored as absent (defensive URL hygiene, no flash), while uuid-shaped-but-unreadable ids flash `prefillUnavailable`.
- Order with >300 lines or a failed line page → commodity left empty (best-effort); order summary itself unreadable → no order prefill + flash.
- Ambiguous commodity across order lines (2+ distinct mapped commodities) → no preselect.
- Annual report year with zero statements → valid empty-shaped report (all zeros/empty arrays), not 404.
- CSV cells: every string cell flows through the shared `serializeExport` (quoting + formula neutralization); test includes a `=HYPERLINK(...)`-shaped supplementary unit.
- Seeding never errors the page: every seeding fetch is wrapped in its own try/catch; any thrown/rejected call resolves to "seed nothing" and the standard create form renders unconditionally — there is no redirect and no error boundary path from seeding.
- Supplier aggregate for a company with no eudr rows → zeros; widget renders null; fetch error → null + debug log.
- Notification service unresolvable (module absent) → no-op log; create throws → rethrow (persistent retry; groupKey dedupes the redelivery); recipient resolution rbac-unavailable → fan-out skipped (fail-closed), logged.
- Risk assessment deleted between event emission and delivery → subscriber no-op log.
- Repeated draft→submitted cycles (submit, back to draft, submit again) → distinct `occurredAt` per transition → distinct notifications (correct); redelivery of one transition → same groupKey → coalesced (correct).
- Undone transition (command undo) → notification already sent stays; informational only (documented).

## Risks & Impact Review

- **Tenant isolation**: both new routes resolve `resolveOrganizationScopeForRequest` and filter tenant + org (batch-1 export precedent); subscribers scope every notification by the event payload's tenant/org (house subscriber contract — payloads originate from our own commands) and the fan-out is org-restricted via the RBAC filter (fail-closed); the supplier widget fetches through the gated aggregate route only.
- **Cascading**: supplier widget is feature-gated + fail-open — customers pages unaffected by eudr errors; notification failures never propagate into commands (post-commit).
- **Residuals**: species advisory only warns (a wood DDS without species is still exportable — deliberate, no-blockers mandate; the chip on statement detail plus the export warning make it hard to miss, and the TRACES side would reject a species-less timber DDS anyway — documented UI copy); annual-report aggregation is SQL-side (bounded); notification fan-out volume bounded by per-org feature-holder count; org-restricted fan-out depends on the RBAC service answering per-candidate checks (bounded by candidate count; skipped entirely on failure).
