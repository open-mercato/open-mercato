# Harness Validation Gate + Module Scaffold Template Fixes

## Summary

AI-assisted standalone module generation was silently producing broken code because:

1. The module-scaffold skill template contained stale API examples that no longer matched the installed `@open-mercato/*` package contracts.
2. The AI harness had no explicit, mandatory validation step to run after scaffolding to catch mistakes before committing.

A malformed `events.ts` generated by the AI crashed `/login` for the entire app, because generated registries import every active module file at startup.

The validation gate is delivered as a **SKILL-level script** documented in the module-scaffold skill (§12 Wire & Verify) — not as a shipped product module or CLI command. The post-scaffold checks compose existing CLI commands (`yarn generate`, `yarn mercato configs cache structural`, `yarn mercato auth sync-role-acls`, `yarn typecheck`) that the harness already exposes; the skill instructs the agent to run them in order rather than wrapping them in a new `@open-mercato/core` module.

## Changes

### Track 1 — Fix stale module-scaffold skill template

**File**: `packages/create-app/agentic/shared/ai/skills/module-scaffold/SKILL.md`

Updated API patterns to match installed packages:

| Pattern | Old (stale) | New (correct) |
|---|---|---|
| API route layout | `api/get/<entity>.ts`, `api/post/<entity>.ts`, … | `api/<entity>/route.ts` (single file) |
| `makeCrudRoute` | `{ entity, entityId, operations, schema }` | `{ metadata, orm, list, create, update, del, indexer }` |
| Route exports | `export default handler` | `export const { GET, POST, PUT, DELETE } = crud` + `export const metadata` |
| `createModuleEvents` | keyed object `{ 'id': { description, payload } }` | `{ moduleId, events: [{id, label, entity, category}] }` |
| `acl.ts` | no default export | `export default features` (required by generated registries) |
| ACL feature IDs | `<module>.view` / `<module>.create` | `<module>.<entity>.view` / `<module>.<entity>.manage` |
| `CrudForm` | `apiPath`, `mode`, `resourceId` props | `onSubmit` with `createCrud`/`updateCrud`, `onDelete` with `deleteCrud` |
| `DataTable` | `apiPath`, `createHref`, `extensionTableId` props | explicit `data`, `isLoading`, `error`, `pagination` via `apiCall` + `useQuery` |
| `ResponseEnricher` | `enrichMany` only | `enrichOne` (required) + `enrichMany` (for N+1 prevention) |

Added mandatory post-scaffold validation gate to §12 Wire & Verify:
```bash
yarn generate
yarn mercato configs cache structural --all-tenants
yarn mercato auth sync-role-acls
yarn typecheck
```

Also updated `packages/create-app/template/AGENTS.md` Mandatory Module Mechanisms table with the same corrected API descriptions.

### Track 2 — Post-scaffold validation gate as a SKILL-level script

The validation gate lives in the module-scaffold skill (`SKILL.md` §12 → Step 5: Run Post-Scaffold Validation Gate) as an explicit, mandatory sequence of existing CLI commands the agent runs after every structural module change — **not** as a new product module or `mercato` CLI subcommand.

```bash
# 1. Re-emit generated registries with the new module
yarn generate

# 2. Purge stale structural cache (nav, module-graph fingerprints)
yarn mercato configs cache structural --all-tenants

# 3. Grant ACL features declared in acl.ts to existing roles
yarn mercato auth sync-role-acls

# 4. Type-check all files — catches API mismatches before they reach runtime
yarn typecheck
```

The gate also keeps the `/login` reachability check as the final manual verification item in §12 → Step 6 (a malformed `events.ts` or missing `acl.ts` export crashes every page, so confirming `/login` still loads catches the #2209 blast-radius failure mode).

**Why SKILL-level instead of a CLI module** (maintainer decision on PR #2216):

- The gate only composes commands the platform already ships (`yarn generate`, `configs cache structural`, `auth sync-role-acls`, `yarn typecheck`). A wrapper module adds a new contract surface, a new module-registry entry, and ACL/feature considerations without adding behavior the agent could not already run directly.
- Scaffolding guidance belongs to the skill that does the scaffolding. Keeping the gate in `SKILL.md` keeps the validation steps next to the patterns they validate and lets them evolve without a package release.
- Standalone apps get the same gate through the generated agentic skill copy — no extra `@open-mercato/core` module needs to ship to every consumer.

## Gaps (acknowledged, out of scope)

- **Dev DB migration guard**: detecting an existing table with a missing migration row and suggesting additive SQL. `yarn db:migrate` remains always user-confirmed per project conventions.
- **Programmatic module shape validation**: static pre-check of exports without running the TS compiler. The `yarn typecheck` step in the gate catches these errors.
- **acl/setup alignment check**: surfacing features declared in `acl.ts` but not granted in `setup.ts` before syncing. The skill's self-review checklist asks the agent to verify `setup.ts` grants every `acl.ts` feature; `yarn mercato auth sync-role-acls` (always run, safe and idempotent) reconciles existing tenants regardless.
- **Sidebar discoverability check**: the structural cache purge in step 2 handles stale nav state indirectly.
