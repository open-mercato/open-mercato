# Warranty & RMA Claims Desk ŌĆö Competitive Refinements (Round 3)

> **Parent specs:** [`2026-07-03-warranty-rma-claims-desk.md`](./2026-07-03-warranty-rma-claims-desk.md) (foundation) and [`2026-07-05-warranty-rma-claims-desk-completion.md`](./2026-07-05-warranty-rma-claims-desk-completion.md) (world-class completion, all 13 phases shipped). This spec is the third round: a 19-product competitive benchmark (July 2026) mapped every remaining gap; the four highest-value, lowest-blast-radius refinements are implemented here, the rest are recorded as an explicit roadmap so future rounds don't re-research the market.

## TLDR
**Key Points:**
- **Resolution execution is the #1 competitive gap.** Every ERP-grade competitor executes financial/fulfillment outcomes from the RMA (D365 replace-dispositions auto-create replacement sales orders; NetSuite's Refund button emits the credit memo; Odoo's Return-for-Exchange spawns the outbound transfer; Saleor's `replace: true` creates the draft replacement order). Our claim desk records `salesReturnId`/`replacementOrderId` as *links only*. Phase 3 ships the first execution bridge: **create a sales return from approved claim lines** with one command (soft-optional `sales` peer).
- **Close two correctness gaps competitors treat as table stakes:** the documented-but-unwired **reverse-tracking auto-advance** (`awaiting_return ŌåÆ received` when the return shipment is delivered ŌĆö Phase 7 of the completion spec shipped the label seam but never the inbound bridge), and **over-claim quantity guards** (D365 "validates that return orders aren't created for a quantity that exceeds the quantity sold"; we accept `qtyClaimed = 999` against a 1-unit order line without blinking).
- **Return-window eligibility** for non-warranty claim types (Shopify return rules, D365 return item policies, Sylius community plugin time-windows): a tenant-configurable `returnWindowDays` that surfaces an `outside_return_window` risk signal ŌĆö advisory, never auto-deny, per the module's established adjudication philosophy.
- Everything else the benchmark surfaced (replacement-order bridge, credit-memo bridge, guest tracking, analytics insights page, repair orders with parts/labor, instant exchange, catalog exchange, RTV consolidation, recalls, extended-warranty plans) is triaged in the **Roadmap** section with effort/dependency notes ŌĆö deliberately NOT in scope here.

**Scope boundary:** all four phases are additive, single-module (plus one narrowly-scoped cross-module *read* and one command *dispatch* into `sales` through the sanctioned soft-optional mechanisms). No new claim statuses, no new ACL features, no changes to any FROZEN surface. One additive migration (one nullable settings column).

**Scope cohesion note (spec-writing gate):** the four phases are independently shippable refinements to one module, bundled per this module's established convention (the completion spec shipped 13 phases the same way). Each phase leaves the app working and can be dropped without affecting the others.

## Problem Statement

The module is feature-complete per the completion spec and ahead of most competitors on receiving/grading, SLA engineering, risk signals, and AI assistance. A 19-product benchmark (returns SaaS: Loop, AfterShip, ReturnGO, Narvar, Happy Returns, Redo, ReturnLogic, Rich Returns; warranty platforms: Extend, Cover Genius/XClaim, Mulberry, OnPoint, Tavant, PTC iWarranty, Pega Warranty, Registria; ERP/commerce: Odoo, D365 SCM, NetSuite, SAP B1, Shopify, Adobe Commerce, Sylius, Saleor, Medusa) shows the desk stops exactly where competitors keep going:

1. **A resolved claim produces no documents.** Staff approve a return, grade the units, pick `credit` dispositions ŌĆö and then leave the module to hand-create the sales return. The `salesReturnId` picker asks the operator to link a document the system itself should have created. (D365: disposition codes "drive both the physical and financial outcome"; NetSuite: "click Refund on the approved authorization".)
2. **The return leg is half-automated.** We generate/record outbound return labels (Phase 7) and persist `return_tracking_number`, and `shipping_carriers` already emits `shipment.status_changed`/`delivered` events ŌĆö but nothing listens. The completion spec explicitly promised "Reverse-tracking webhook ŌĆ” can auto-advance `awaiting_return ŌåÆ received`" and never wired it.
3. **Quantity honesty is unenforced.** `claim_line.create` validates `qtyApproved Ōēż qtyClaimed` but never `qtyClaimed` against what the linked order line actually sold ŌĆö neither on one line nor cumulatively across claims (multi-claim double-dipping is a top fraud pattern; D365 POS validates "available to return" server-side for exactly this reason).
4. **Time-based eligibility exists only for warranty.** The entitlement resolver answers "is this unit in warranty?" but a plain `return` claim has no notion of a merchant return policy window ŌĆö table stakes in every returns SaaS (Loop zones/windows, AfterShip warranty-period rules, Shopify return rules).

## Proposed Solution ŌĆö four phases

### Phase 1 ŌĆö Reverse-tracking auto-advance (completes completion-spec Phase 7)

New persistent subscriber `subscribers/return-shipment-tracking.ts` on **`shipping_carriers.shipment.delivered`** ŌĆö the carrier framework's dedicated terminal event (declared in `packages/core/src/modules/shipping_carriers/events.ts`, emitted with the same payload as `status_changed` by `lib/shipping-service.ts` and `workers/webhook-processor.ts`), so the subscriber does not re-derive terminality *(jury F-26)*. Peer referenced by event-id string only ŌĆö identical soft-optional pattern to `email-to-claim` on `inbox_ops.email.received`; no import of the peer module.

- Payload contract (verified): `{ shipmentId, providerKey, previousStatus, newStatus, organizationId, tenantId }`.
- Handler: load the shipment row by id via scoped kysely (`to_regclass` guard so a standalone app without the carrier tables no-ops cleanly; a missing/soft-deleted shipment row is an explicit no-op *(jury F-24)*) to obtain `tracking_number`; find claims with `status = 'awaiting_return' AND return_tracking_number = <tracking> AND deleted_at IS NULL` in the same tenant/org. **Advance only on an unambiguous match**: exactly one claim ŌåÆ dispatch `warranty_claims.claim.transition` (`toStatus: 'received'`) with a system-actor context; two or more matches ŌåÆ warn-log with the claim numbers and skip (staff advance manually) *(jury F-08)*.
- Optimistic lock: the system dispatch carries no expected-version header, and `enforceWarrantyClaimOptimisticLock` is a no-op without one (established behavior, `commands/shared.ts` ŌĆö same contract the comment command documents), so no version token is needed *(jury F-06 refuted with evidence)*.
- Error policy: business no-ops (no match, wrong status, peer absent) are swallowed with a debug log; **infrastructure errors rethrow** so the persistent queue's bounded retry policy re-delivers ŌĆö a transient DB blip cannot silently eat the auto-advance, and a poison event exhausts retries without wedging the pipeline *(jury F-01)*.
- Timeline: the transition input schema gains an **additive optional `systemNote` key** that the command records in the `status_changed` timeline event payload; the subscriber passes `warranty_claims.timeline.autoReceivedFromTracking` so the desk shows *why* the claim advanced (`transitionClaimInputSchema` is `.strict()` today with no note carrier ŌĆö verified *(jury F-27)*). `systemNote` is **constrained by zod to a `warranty_claims.`-prefixed i18n key pattern** ŌĆö never free text ŌĆö so nothing untranslated or sensitive can ride the (customer-visible) `status_changed` payload and the free-textŌåÆencryption invariant is not in play *(jury r2)*. Two concurrent delivered events race benignly: the second dispatch hits the state machine's invalid-transition guard and is swallowed as a business no-op *(jury r2)*.
- Lookup index: the round's single migration (filed under Phase 4 to keep one migration per round) adds a partial index on `warranty_claims (tenant_id, organization_id, return_tracking_number) WHERE return_tracking_number IS NOT NULL AND deleted_at IS NULL` *(jury F-23)*. Phase 1 works correctly without the index (tenant-scoped scan on a rare event); the index is an optimization, so the phases stay independently droppable *(jury r2)*.
- Multi-parcel note: the claim model carries one `return_tracking_number`; delivery of that parcel is the receiving trigger, and the receiving workbench (grades + `qtyReceived`) remains the authority on what actually arrived *(jury F-18)*.
- No settings flag: matching requires an exact tracking-number + status match that only the label flow itself creates. (Manual-entry tracking numbers get the same benefit for free.)

### Phase 2 ŌĆö Quantity honesty guards (over-claim)

Two-tier design mirroring the module's hard-gate-vs-signal philosophy:

1. **Hard gate (per claim vs sold quantity):** a shared helper `assertClaimedQtyWithinSold` applied on **every line-write path** ŌĆö `claim_line.create`, `claim_line.update`, **and the inline `lines` array of `claim.create`** (which the desk create page, portal intake, external API, and email bridge all flow through ŌĆö the gate cannot live only in the line commands *(jury F-04/F-13)*). When a line carries `orderLineId`, fetch the sold quantity via the existing `querySalesReferenceRow` scoped lookup (`commands/claims.ts` ŌĆö extend the `sales_order_lines` field list with `quantity`; degrades to no-check when `sales` is absent or the row/qty is unavailable ŌĆö same as today's reference validation). The gate sums `qtyClaimed` across **all non-deleted lines of the same claim referencing the same `orderLineId`** (duplicate-row evasion inside one claim is blocked *(jury F-05)*); if the claim's total for that order line exceeds the sold quantity ŌåÆ 400 `warranty_claims.errors.qtyExceedsOrdered` (├Ś4 locales). This cannot break TC-WC-001..013: every existing fixture claims within the ordered quantity.
2. **Risk signal (cumulative across claims):** new `over_quantity_claim` factor in `lib/risk.ts` ŌĆö for each line with `orderLineId`, sum `qty_claimed` of non-deleted lines on other non-cancelled claims (tenant/org-scoped, `rejected` lines excluded) referencing the same `orderLineId`; when `existing + current > soldQty`, emit a medium-severity signal (high when Ōēź 2├Ś sold). Advisory, consistent with `duplicate_order_claim`. Risk-high still forces manual review through the existing adjudication path ŌĆö no new deny behavior. **Deliberately not a hard "remaining returnable quantity" gate** *(jury F-20)*: rejected/appealed history and legitimate re-claims (repair failed, second fault) make a cross-claim hard block wrong more often than right; the signal + manual review is the module's established answer.

### Phase 3 ŌĆö Resolution execution: create a sales return from the claim

The first execution bridge, scoped to the lowest-risk document (a sales return moves no money; it is the receiving/fulfillment document `sales` already owns).

- **Command** `warranty_claims.claim.create_sales_return` (new, in `commands/claims.ts`):
  - Guards: claim exists in scope; `orderId` present; status Ōłł `approved | awaiting_return | received | inspecting`; `salesReturnId` empty (duplicate-safe ŌĆö mirrors `vendorClaimLineId` guard); optimistic lock enforced on the **claim**; at least one eligible line.
  - Eligible lines: non-deleted, `lineStatus Ōłł approved|received|inspected|resolved`, `orderLineId` present, effective qty = `qtyApproved ?? qtyClaimed` > 0. Lines without `orderLineId` are skipped and reported back (`skippedLineIds`) ŌĆö the operator sees an explicit partial-coverage notice, matching the module's honest-degradation style.
  - Executes the sales module's own command `sales.returns.create` via `commandBus` (verified contract in `packages/core/src/modules/sales/commands/returns.ts` + `data/validators.ts:833`: `{ orderId, reason?, notes?, returnedAt?, lines: [{ orderLineId, quantity }] } ŌåÆ { returnId }` *(jury F-25)*), reason = the claim number itself (locale-neutral ŌĆö commands run without a request locale ŌĆö and it doubles as the orphan-traceability marker, below). Sales-absent ŌåÆ 400 `warranty_claims.errors.salesUnavailable`.
  - **Header scrubbing (jury F-14):** `sales.returns.create` enforces the order-level optimistic lock from the propagated request (`returns.ts:656`); the caller's expected-version header carries the *claim's* token, so the bridge dispatches the nested command with a context whose request omits the optimistic-lock header ŌĆö the same per-child-override rule the root `AGENTS.md` mandates for forms that mutate other entities, applied server-side.
  - **Shipped-quantity cap (jury F-15):** `sales.returns.create` hard-caps each line at its *shipped* quantity (`loadShippedQuantityByLine`, `returns.ts:584/680`). The bridge translates that rejection into 400 `warranty_claims.errors.salesReturnQuantityRejected` naming the offending lines; the operator resolves by shipping/adjusting in sales or lowering `qtyApproved`. TC-WC-030's fixture must create shipments for the order lines before exercising the bridge.
  - **Concurrency (jury F-02):** after the nested create returns, the stamp is written in a short transaction that re-loads the claim with `LockMode.PESSIMISTIC_WRITE` (the `create_vendor_recovery` pattern) and re-checks `salesReturnId IS NULL`; a lost race compensates by dispatching `sales.returns.delete` for the just-created return and answering 400 `alreadyLinked`. Check-then-act without the lock is explicitly not enough.
  - **Failure story (jury F-03/F-10):** if the stamp transaction fails after the sales return exists, the command dispatches the compensation delete; if *that* also fails, the command surfaces a 500-class error whose log line and claim timeline entry carry the orphaned `returnId` ŌĆö and the sales return's reason text embeds the claim number, so the orphan is discoverable from either side. No silent success in any branch.
  - On success: stamp `claim.salesReturnId = returnId`, append a `system` timeline event (`warranty_claims.timeline.salesReturnCreated`, customer-invisible), emit the existing claim CRUD-updated side effect (cache/index refresh ŌĆö the lesson from the comment-command cache bug).
  - **Undo:** deletes the created sales return via `sales.returns.delete` **passing the sales return's version token captured right after creation** (`sales.returns.create` returns only `{returnId}`, so the bridge reads the return's `updated_at` via the scoped lookup and stores it in the undo payload; the undo compares it against the return's current `updated_at` via the scoped lookup and aborts loudly (409) on mismatch or when no token was captured, deleting only on an exact match ŌĆö same guarantee as header injection with the check made explicit *(jury F-28, r2, impl-council F-07/F-08)*) and clears the stamp ŌĆö mirroring the vendor-recovery-undo reconciliation pattern. Compensation/undo delete under the route's `sales.returns.create` grant is deliberate: unwinding the exact document that grant just created is within the granted capability's intent *(jury r2)*.
- **API** `POST /api/warranty_claims/sales-return` `{ id }` ŌåÆ **200** `{ salesReturnId, skippedLineIds }` ŌĆö `requireAuth` + features `warranty_claims.claim.manage` **and `sales.returns.create`** (the command bus does not re-check the peer module's RBAC, so the route must ŌĆö a warranty-only operator must not mint sales documents *(jury F-09)*), mutation-guard wired, openApi entry, optimistic-lock header honored for the claim (same shape as `return-label`).
- **UI**: a "Create sales return" action in the detail page's fulfillment section (next to the existing sales-return *picker*), visible when `orderId` set Ōł¦ status eligible Ōł¦ `salesReturnId` empty; on success flashes (with a skipped-lines notice) and reloads so the linked return shows on the claim ŌĆö a deep-link into the return itself rides the roadmap with the replacement-order item (sales returns render inside the order detail, not on a standalone route). Guarded via `useGuardedMutation` (page is non-`CrudForm` for actions). Lines whose effective quantity is fractional are reported in `skippedLineIds` ŌĆö `sales.returns.create` accepts whole units only (`returnLineQuantitySchema` is integer-min-1) *(impl-council r2 F-08)*.

### Phase 4 ŌĆö Return-window eligibility policy

- **Settings:** new nullable int column `warranty_claim_settings.return_window_days` (additive migration + snapshot). `null` = feature off (default ŌĆö zero behavior change). Settings UI: number field beside the SLA block, localized help text.
- **Evaluation:** applies to `claimType Ōłł return | core_return` with an `orderId`. Order date anchor = `sales_orders.placed_at ?? created_at` (`placed_at` verified at `sales/data/entities.ts:415`; the placement date is the business anchor, not the row-insert timestamp *(jury F-11)*) fetched through the same scoped lookup used by `querySalesReferenceRow` (extend field list; degrade to no-signal when unavailable). Anchoring on *delivery/fulfillment* date (the Shopify/D365 refinement) is deliberately deferred to the roadmap ŌĆö it needs shipment-date resolution across optional fulfillment data; the placement anchor is stricter (flags earlier), which is safe because the signal is advisory *(jury F-17/F-19)*. At submit and in the risk endpoint: when `now ŌłÆ anchor > returnWindowDays` ŌåÆ new risk signal `outside_return_window` (medium; high when > 2├Ś window). Surfaces in the existing risk panel and ŌĆö because risk-high forces manual review ŌĆö automatically keeps out-of-window claims away from auto-approval. **No hard deny**: B2B phone intake stays flexible; the merchant reviews. **Staff-side only in this round**: the risk endpoint is staff-authed, so the portal intake hint requires a customer-scoped endpoint ŌĆö moved to the roadmap rather than half-wired *(jury F-07)*.

## Edge Cases & Failure Scenarios

- **Tracking number reused across tenants/claims:** matching is tenant/org-scoped and status-gated; an ambiguous match (two or more `awaiting_return` claims sharing one tracking number) advances nothing and warn-logs the claim numbers for manual receiving.
- **Shipment table absent (standalone app without carrier module):** `to_regclass` guard ŌåÆ subscriber no-ops; unit-tested absent-module path.
- **Sales return creation succeeds but stamp fails ŌĆö canonical sequence *(jury r2)*:** (1) validate + optimistic-lock the claim outside any transaction; (2) dispatch `sales.returns.create` (no surrounding claim transaction ŌĆö the sales command owns its own); (3) read the created return's `updated_at` via the scoped lookup (undo token); (4) short `PESSIMISTIC_WRITE` transaction: reload claim, re-check `salesReturnId IS NULL`, stamp + timeline event, flush; (5) on step-4 failure or lost race, compensate by dispatching `sales.returns.delete` and surface a clean error. If a compensation delete itself fails, the loud error + timeline entry carry the orphaned `returnId` and the sales return's reason embeds the claim number; a retry after an uncompensated failure is an operator decision made with that evidence visible ŌĆö accepted residual risk *(jury r2)*.
- **Order line partially fulfilled / qty lookup returns null:** hard gate skips (no data, no block); risk signal skips. Honest degradation over false 400s.
- **`returnWindowDays` set while sales absent:** no order date ŌåÆ no signal; settings field stays editable (documented in help text).
- **Concurrent create_sales_return double-click:** optimistic-lock header + `salesReturnId`-empty guard; the second request 400s (`alreadyLinked`).

## Risks & Impact Review

- **Blast radius: low-medium.** One nullable column; one new command + endpoint + subscriber; two risk factors; validator tightening on one field path. The hard gate (Phase 2.1) is the only behavior change to an existing path ŌĆö bounded to lines that carry `orderLineId` AND a resolvable sold quantity, and provably compatible with all existing integration fixtures.
- **Cross-module writes:** only via `sales.returns.{create,delete}` command dispatch ŌĆö the sales module's own validation/side effects run; no warranty code touches sales tables directly.
- **Event-pipeline safety:** the tracking subscriber swallows business no-ops and rethrows only infrastructure errors, so the persistent queue's bounded retry handles transients and a poison event exhausts retries without wedging the shipping pipeline *(consistent with Phase 1's reconciled error policy)*.
- **Phase 2 behavior change (explicit):** the sold-qty hard gate is the round's only tightening of an existing write path ŌĆö scoped to lines carrying `orderLineId` with a resolvable sold quantity, verified compatible with every existing fixture *(jury r2)*.
- **Rollback:** each phase is independently revertible; migration `down()` restores the prior schema; feature default-off (Phase 4) and status-gated automation (Phase 1) mean a revert has no data cleanup.
