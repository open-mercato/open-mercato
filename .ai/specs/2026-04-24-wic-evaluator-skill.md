## TLDR

We are adding a `wic-evaluator` skill that packages two existing WIC tools into a local, in-repo workflow:
- `wic_data_fetcher.mjs` for fetching the raw GitHub dump
- `WIC Assessment Guide` as the single source of truth for the scoring rubric

The skill supports two modes of operation:
- ordinary WIC scoring from a live dump
- determinism testing against a frozen fixture

Status for the first PR: testing phase — packaged for in-repo replay-testing and rubric iteration; not yet an authoritative production scorer.

## Overview

The PRM describes the contract for importing and storing `WIC`, but the scoring itself happens outside the module. This skill moves the operational scoring workflow into the repo as a reproducible bundle of instructions, references, and scripts.

## Problem Statement

Until now, WIC rules were scattered between an external guide and a fetcher script. That made it hard to:
- run scoring consistently across developers
- perform a fair determinism test against an identical input
- audit whether the model actually applies the rubric 1:1

## Proposed Solution

Add a skill at `.ai/skills/wic-evaluator/` containing:
- `SKILL.md` describing the scoring and replay-test workflow
- `references/wic-assessment-guide.md` as the local rubric
- `scripts/wic_data_fetcher.mjs` as a local copy of the fetcher
- `scripts/freeze_fixture.sh` for saving a frozen dump
- `scripts/compare_reports.mjs` for comparing multiple reports generated from the same fixture

## Architecture

### Inputs

- GitHub profiles
- date range `from/to`
- GitHub repo, defaulting to `open-mercato/open-mercato`
- optional frozen markdown fixture

### Flow

1. `freeze_fixture.sh` runs `wic_data_fetcher.mjs` and saves the markdown dump.
2. The skill reads `references/wic-assessment-guide.md`.
3. The model scores the dump against the rubric and returns a single markdown table.
4. For determinism tests, multiple reports generated from the same fixture are compared via `compare_reports.mjs`.

### Boundaries of Responsibility

- the fetcher gathers data and does not compute any score
- the guide defines the rubric and the output format
- the skill orchestrates the workflow
- the comparator checks consistency of results across runs

## Data Models

### Fixture dump

A markdown file generated by the fetcher, containing:
- range metadata
- a list of active bounties
- a list of contributions with PR/issue details, stats, and comments/reviews

### WIC report

A markdown table with the columns:
- `Person`
- `GH profile`
- `Month`
- `WIC script version`
- `WIC Score`
- `WIC Level`
- `Bounty bonus`
- `Why bonus`
- `What we included and why?`
- `What we excluded and why?`

## API Contracts

No runtime platform API is added. The skill provides these local CLI contracts:

- `node .ai/skills/wic-evaluator/scripts/wic_data_fetcher.mjs --profiles <logins> --from <YYYY-MM-DD> --to <YYYY-MM-DD> --format markdown`
- `.ai/skills/wic-evaluator/scripts/freeze_fixture.sh --profiles <logins> --from <YYYY-MM-DD> --to <YYYY-MM-DD> --output <path>`
- `node .ai/skills/wic-evaluator/scripts/compare_reports.mjs <report-a.md> <report-b.md> [...]`

## Risks & Impact Review

### Main risks

- The guide uses soft concepts like "huge" or "large scope", so the model may drift on replay.
- A live GitHub fetch is not, on its own, a determinism test, because the input itself changes.
- The fetcher depends on `gh`, so the local environment must be authenticated, and different `gh` versions may differ slightly in response shape (gh version drift).
- GitHub API responses are paginated — without pagination, an active contributor easily exceeds the default `per_page=30` and the dump becomes incomplete (a determinism killer). Mitigated by `--paginate` / `--limit 500` on every `gh` call.
- The markdown dump must be full — truncating PR/Issue bodies to 200/500 characters would strip spec-id references and break the deterministic grouping rule. Mitigated by printing bodies and comments in full inside code blocks.
- For exact replay the `json` format is recommended, because markdown is sensitive to cosmetic rendering changes.

### Mitigations

- The skill clearly separates `live scoring` from `frozen replay`.
- The comparator measures both exact match and scoring-field consistency.
- The guide is copied locally into the repo to reduce source drift.
- The guide tightens deterministic grouping, `Accepted Bug Report`, and the monthly `WIC Level` rule to reduce drift between runs.
- The fetcher paginates through the GitHub API and records comments/reviews with `authorAssociation` and `createdAt`, so the Core Team approval rule has an explicit signal in the dump.
