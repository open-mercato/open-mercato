## Executive Summary

The spec is **ready to implement** with a small set of bring-up verifications. It is **low-risk for backward compatibility**: it adds a greenfield `evals/` tree plus one CI workflow and consumes the *published* `@open-mercato/*` packages and `create-mercato-app` as a black box — it touches **zero** of the 13 contract surfaces. The real risks are operational (standalone-scaffold gotchas, seeded-admin identity, Daytona networking, full-stack verifier runtime), all of which have known mitigations drawn from `.ai/lessons.md`. The one load-bearing technical assumption — that a flat `/api/bookmarks` is achievable — has been **verified in the generator source**.

**Recommendation: Ready to implement** (Phase 1), addressing the "Before Implementation" items as bring-up checks rather than blockers.

## Risk Assessment

### High Risks

| Risk | Impact | Mitigation |
|------|--------|-----------|
| **Seeded-admin identity mismatch.** The hidden Playwright spec authenticates as `admin@acme.com/secret` (the `@open-mercato/core` integration helper default). If the scaffold's `mercato init` seeds a different admin email/password, every FAIL_TO_PASS auth call fails — a false negative that looks like agent failure. | All runs falsely fail. | At bring-up, run the baseline setup and confirm the seeded credentials. If different, set the eval-app `.env`/init flags to seed `admin@acme.com/secret`, or override the helper via `BASE_URL`/credential env. Add an assertion in `app-runner.ts` that login succeeds before running the suite. |
| **Verifier runtime budget on Daytona.** Full OM build + migrate + init + `next start` + Playwright can exceed default timeouts. | Spurious timeouts. | Set generous `[agent].timeout_sec` / `[verifier].timeout_sec`; cache nothing but measure a baseline oracle run on `-e docker` first; tune. |

### Medium Risks

| Risk | Impact | Mitigation |
|------|--------|-----------|
| **Immutable lockfile (YN0028).** First `yarn install` in a fresh scaffold fails on CI defaults. | Setup fails. | Set `YARN_ENABLE_IMMUTABLE_INSTALLS=0` for the eval-app install step (lessons.md). |
| **Daytona default network restriction.** `npx create-mercato-app`, `yarn install`, and agent model API calls need outbound. | Setup/agent fail. | Allow network via `[environment].network_mode`/Daytona `HARBOR_NETWORK`; confirm at bring-up. |
| **Webhook/security env contract.** If `mercato init`/setup or any scaffold test path touches gateway/webhook code, missing `MOCK_GATEWAY_WEBHOOK_SECRET` causes 401s. | Setup/test flake. | Use `--preset empty` (no checkout/gateway demo) and mirror the full env contract from `scripts/test-create-app-integration.ts` if needed. |
| **Standalone entity-id generation.** Route imports `E.bookmarks.bookmark` from `#generated`. For app-local modules the id is generated from source; confirm it appears. | Build/typecheck fail. | App module is `.ts` source, so the source-shape generator path applies (not the dist path that broke package entities). Verify `E.bookmarks.bookmark` exists after `yarn generate` in the oracle run. |
| **`yarn generate` no-diff PASS_TO_PASS.** Structural cache post-step touches `*.generated.*` mtimes (not content). | A naive `git diff` could flag mtime-only / content-identical churn. | Compare by **content** (`git diff --quiet` ignores mtime; git tracks content), and run `yarn generate` twice in setup so the baseline is already settled before the agent. |

### Low Risks

| Risk | Impact | Mitigation |
|------|--------|-----------|
| Codex `gpt-5.5` exact model ID | Agent misconfig | Confirm against pinned Codex CLI; store in `task.toml`. |
| esbuild bundling of `verify.mjs` from `evals/lib` | Build step complexity | Keep `lib` dependency-light; bundle to a single file copied into `tests/`. |
| ts-morph parsing variance across agent code styles | Rubric false negatives | Write checks against *semantics* (import specifier, decorator presence) not formatting; add oracle + negative-control fixtures. |

## Remediation Plan

### Before Implementation (Must Do — bring-up checks, not blockers)

1. **Baseline a scaffold locally**: `npx create-mercato-app@<current> eval-app --preset empty` → wire `.env` to local pg/redis/meili → `YARN_ENABLE_IMMUTABLE_INSTALLS=0 yarn install` → `yarn generate && yarn db:migrate && yarn mercato init && yarn build`. Capture the seeded admin credentials and confirm `/api/bookmarks` is reachable after adding the module.
2. **Confirm the integration helper** `@open-mercato/core/helpers/integration/api` is present in the published core and its default credentials match the seeded admin.
3. **Confirm Daytona network + compose** with a trivial run before investing in the full verifier.

### During Implementation (Add to Spec)

1. Add `YARN_ENABLE_IMMUTABLE_INSTALLS=0`, network allowance, and (if needed) the webhook env contract to the spec's setup sequence.
2. Specify the PASS_TO_PASS `yarn generate` check as **content-based**, with a double-generate in baseline.
3. Promote negative-control fixtures to explicit Phase 3 deliverables (one per criterion).
4. Note the optimistic-locking default-ON behavior so the rubric/judge does not penalize the agent for `updated_at` in responses.

### Post-Implementation (Follow Up)

1. Build `trend.mjs` once S3 records accumulate across ≥2 releases.
2. Revisit Harbor Hub push if non-interactive auth is confirmed.
3. Remove the temporary `pull_request` CI trigger after the first green real-agent run.
