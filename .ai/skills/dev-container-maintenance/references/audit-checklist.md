# Dev Container Audit Checklist

Run each check by reading the source files and comparing values. Report all mismatches.

## 1. Version Alignment

Read these file pairs and compare versions:

| Check | Regular source file | Dev container file | What to compare |
|-------|--------------------|--------------------|-----------------|
| Node.js | `.nvmrc` | `.devcontainer/Dockerfile` | `.nvmrc` content vs `FROM node:<ver>-slim` major version |
| Node.js (engines) | `package.json` `engines.node` | `.devcontainer/Dockerfile` | Engine range vs image major version |
| Yarn | `package.json` `packageManager` | `.devcontainer/Dockerfile` | Exact version in `corepack prepare yarn@<ver>` |
| PostgreSQL | `docker-compose.yml` postgres `image:` | `.devcontainer/docker-compose.yml` postgres `image:` | Image tags must match |
| Redis | `docker-compose.yml` redis `image:` | `.devcontainer/docker-compose.yml` redis `image:` | Image tags must match |
| Meilisearch | `docker-compose.yml` meilisearch `image:` | `.devcontainer/docker-compose.yml` meilisearch `image:` | Image tags must match |

**Action on mismatch**: Update the dev container file to match the regular source.

## 2. Service Parity

Compare services required for development:

1. Read `docker-compose.yml` (root) — list all infrastructure services (ignore app/verdaccio/opencode)
2. Read `.devcontainer/docker-compose.yml` — list all services except `workspace`
3. Any infrastructure service in root but missing from devcontainer needs to be added

For each service, verify:
- Image tag matches
- Port matches
- Health check exists
- Environment variables match (credentials, config)

**Also check** `docker-compose.fullapp.dev.yml` if it exists — it may reference services not in the base `docker-compose.yml`.

## 3. Environment Variable Sync

This is the most common source of drift. Three-step check:

### 3a. Find new keys in `.env.example`
1. Read `apps/mercato/.env.example` — extract all variable names (lines matching `^[A-Z_]+=` or `^# [A-Z_]+=`)
2. Read `.devcontainer/scripts/setup-env.sh` — extract all variable names referenced in `sed` commands
3. New keys in `.env.example` that reference `localhost` or need container-specific values require new `sed` rules in `setup-env.sh`

### 3b. Verify hostname rewrites are correct
Check that `setup-env.sh` rewrites cover all `localhost` references that should point to container service names:
- `localhost:5432` or `localhost` in DATABASE_URL → `postgres`
- `localhost:6379` or `localhost` in REDIS_URL → `redis`
- `localhost:7700` or `localhost` in MEILISEARCH_HOST → `meilisearch`
- `localhost:8200` or `localhost` in VAULT_ADDR → `host.docker.internal` (reaches host)

### 3c. Check forwarded env vars
Read `devcontainer.json` `remoteEnv` section. Verify all API keys that developers need inside the container are forwarded from the host (e.g., `ANTHROPIC_API_KEY`, `OPENAI_API_KEY`). If `.env.example` references new external API keys, consider adding them to `remoteEnv`.

## 4. Package Coverage (dist/ volumes)

1. List all directories in `packages/*/` — these are the monorepo packages
2. Read `.devcontainer/docker-compose.volumes.yml` — list all `pkg_*_dist` volume entries
3. Each package should have a corresponding `pkg_<name>_dist` volume (hyphens replaced with underscores)

**Note**: This file is auto-generated by `generate-compose-volumes.sh`. If packages are missing, the script itself may need fixing. Read the script and verify its package scanning logic matches the actual `packages/` directory structure.

## 5. System Dependencies

1. Read `.devcontainer/Dockerfile` — list all `apt-get install` packages
2. Check if the project has added new native dependencies. Look for:
   - `package.json` `optionalDependencies` with native addons (e.g., `sharp`, `canvas`, `bcrypt`)
   - New `*.node` or `.so` references in build errors
   - `node-gyp` compilation requirements in any package
3. Verify `python3`, `make`, `g++` are present (needed for native Node.js addons)
4. Verify `postgresql-client` is present (needed for `psql` access to container DB)

## 6. Port Forwarding

1. Read `.devcontainer/docker-compose.yml` — extract all exposed ports from services
2. Read `.devcontainer/devcontainer.json` `forwardPorts` array
3. Every service port should appear in `forwardPorts` with appropriate `portsAttributes` (label + onAutoForward)
4. **Non-workspace services MUST use named service syntax** — numeric entries (e.g., `5432`) are forwarded on the workspace container, not on the service that actually listens on that port. This breaks host tool connections.

Standard ports and required `forwardPorts` entries:
- `3000` — App (Next.js, workspace service) — numeric OK — `onAutoForward: "notify"`
- `"postgres:5432"` — PostgreSQL — named syntax required — `onAutoForward: "silent"`
- `"redis:6379"` — Redis — named syntax required — `onAutoForward: "silent"`
- `"meilisearch:7700"` — Meilisearch — named syntax required — `onAutoForward: "silent"`

**Action on mismatch**: Replace numeric entries for non-workspace services with `"<service>:<port>"` named syntax.

## 7. Build Sequence

Compare the bootstrap chain:

1. Read `.devcontainer/scripts/post-create.sh` — list all steps in order
2. Read `docker/scripts/dev-entrypoint.sh` — list its startup steps
3. Read root `package.json` scripts: `initialize`, `dev:greenfield`, `build`, `build:packages`, `generate`
4. Verify `post-create.sh` covers the same steps as `yarn dev:greenfield` or `yarn initialize`

Expected sequence in `post-create.sh`:
```
[0] Fix volume permissions (sudo chown)
[1] Generate .env (setup-env.sh)
[2] yarn install
[3] Install skills
[4] yarn build:packages (first pass)
[5] yarn generate
[6] yarn build:packages (second pass)
[7] yarn mercato init OR yarn db:migrate
```

If `package.json` scripts have changed (new build steps, reordered dependencies), `post-create.sh` must be updated to match.

## 8. Lifecycle Hook Consistency

Verify `devcontainer.json` hooks point to the correct scripts:

```
initializeCommand  → .devcontainer/scripts/generate-compose-volumes.sh
postCreateCommand  → .devcontainer/scripts/post-create.sh
postStartCommand   → .devcontainer/scripts/post-start.sh
```

Verify each script exists and is executable (`chmod +x`).

## 9. VS Code Extensions

1. Read `.devcontainer/devcontainer.json` `customizations.vscode.extensions` array
2. Check if the project has added new linting, formatting, or tooling configs that require VS Code extensions
3. Common additions: new language support, new framework extensions

## 10. Documentation Sync

Verify that documentation matches the actual dev container implementation.

### `.devcontainer/README.md`
1. Read `.devcontainer/README.md`
2. Cross-check each table against the actual files:
   - **Services table** matches services in `.devcontainer/docker-compose.yml`
   - **Named Volumes table** matches volumes in `docker-compose.yml` + `docker-compose.volumes.yml`
   - **Lifecycle Hooks** section matches `devcontainer.json` hooks and the scripts they call
   - **Troubleshooting table** covers all known failure modes
3. Flag any stale rows (services removed, volumes renamed, steps reordered)

### `.ai/specs/SPEC-050-2026-02-26-dev-container-setup.md`
1. Read the spec
2. Check:
   - **Service Versions table** matches actual image tags in `.devcontainer/docker-compose.yml`
   - **Volume Strategy table** matches actual volumes
   - **Configuration / Forwarded Ports** matches `devcontainer.json`
   - **Changelog** has an entry for every change made since the last audit
3. If the spec is outdated, use the **`spec-writing` skill** (`/spec-writing`) to update it following project conventions

**Action on mismatch**: Update documentation to match implementation. Never leave docs stale after a fix.
