## Prerequisites {#macos}

Install the following tools before continuing. Each step includes a verification command.

### 1. Git

macOS may already have Git via Xcode Command Line Tools. Check with `git --version`. If it is missing, install the CLT:

```bash
xcode-select --install
```

Or install Git via Homebrew (see next step).

### 2. Homebrew

If Homebrew is not already installed:

```bash
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"
```

Follow the post-install instructions printed by the script to add Homebrew to your shell PATH.

### 3. Node.js 24

```bash
# Option A — Homebrew
brew install node@24
echo 'export PATH="/opt/homebrew/opt/node@24/bin:$PATH"' >> ~/.zshrc
source ~/.zshrc

# Option B — nvm (manage multiple Node versions)
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.0/install.sh | bash
# Close and reopen your terminal, then:
nvm install 24
nvm use 24
nvm alias default 24
```

Verify: `node --version` → `v24.x.x`

### 4. Yarn 4.12.0

Corepack ships with Node.js. Enable it once per machine, then activate Yarn:

```bash
corepack enable
corepack prepare yarn@4.12.0 --activate
```

Verify: `yarn --version` → `4.12.0`

### 5. Docker Desktop

Download and install [Docker Desktop for Mac](https://docs.docker.com/desktop/setup/install/mac-install/). Start it after installation.

Verify:

```bash
docker --version
docker compose version
```

## Infrastructure services

Docker Desktop runs PostgreSQL, Redis, and Meilisearch. Start them from the repository root after cloning (step below):

```bash
yarn infra:up
```

This starts:
- **PostgreSQL 17** with the pgvector extension — port `5432`
- **Redis 7** — port `6379`
- **Meilisearch** — port `7700`

---

## Clone and configure

```bash
git clone https://github.com/open-mercato/open-mercato.git
cd open-mercato
git checkout develop
```

Copy the environment template:

```bash
cp apps/mercato/.env.example apps/mercato/.env
```

Edit `apps/mercato/.env` and set at least these three variables:

```env
DATABASE_URL=postgres://postgres:postgres@localhost:5432/open-mercato
JWT_SECRET=change-me-dev-secret
REDIS_URL=redis://localhost:6379
```

Generate a strong `JWT_SECRET`:

```bash
openssl rand -hex 32
```

---

## Install dependencies and bootstrap

```bash
yarn install

yarn build:packages
yarn generate
yarn build:packages   # second run required — compiles generated scripts

yarn initialize
```

`yarn initialize` runs migrations, seeds roles, provisions an admin user, and loads demo CRM data. Add `--no-examples` to skip demo content. The admin credentials are printed at the end.

---

## Start the app

```bash
yarn dev
```

Open **http://localhost:3000/backend** and sign in with the credentials printed by `yarn initialize`.

The compact runtime also serves a splash page at **http://localhost:4000** with live startup progress. Press `d` while `yarn dev` is running to toggle raw log output.

On memory-constrained machines you can lower `yarn dev`'s footprint by narrowing which workspace packages the watcher tracks with `OM_WATCH_SCOPE` (e.g. `OM_WATCH_SCOPE=auto-optimized yarn dev`). The active mode is printed with an emoji at startup. See [Choosing which packages the watcher tracks](../appendix/troubleshooting#choosing-which-packages-the-watcher-tracks-watch-scope) for the full reference.

---

## Upgrading an existing checkout

```bash
git pull
yarn install
yarn db:migrate
yarn generate
yarn dev
```

## One-command setup (recommended) {#windows-one-command}

The fastest path on Windows — including a **completely clean machine with no Git, no Docker, and no Node.js** — is the bundled launcher:

1. Get the repo (clone it, or download the [ZIP](https://github.com/open-mercato/open-mercato/archive/refs/heads/main.zip) and extract it). On a bare machine you can even download just [`start-windows.bat`](https://raw.githubusercontent.com/open-mercato/open-mercato/main/starters/docker/windows/start-windows.bat) — it installs Git and clones the repo for you.
2. Double-click **`starters\docker\windows\start-windows.bat`** (auto-detects the container runtime). Organizations standardized on Rancher Desktop double-click **`start-windows-rancher.bat`** instead; **`start-windows-docker.bat`** forces Docker Desktop.

**Hardware:** 16 GB RAM recommended, 12 GB minimum (the launcher checks and refuses below 12 GB — `-SkipResourceCheck` overrides); ~20 GB free disk for the first run. On machines under 20 GB RAM the launcher also writes a WSL2 memory cap into `%USERPROFILE%\.wslconfig` (never touching an existing `memory=` line) so the container VM neither OOMs (Windows 11 default: 8 GB cap) nor starves the host (Windows 10 default: 80% of RAM) — see the [WSL2 guide](./wsl2) for manual tuning. Unsure whether a locked-down machine will cooperate? Run **`starters\docker\windows\check-windows.bat`** first — a read-only preflight (policies, WSL2, proxy/TLS interception, download hosts, antivirus, ports) ending in a READY / CAVEATS / LIKELY-BLOCKED verdict.

The launcher installs missing prerequisites (Git, WSL2, Docker Desktop — **no Node.js or Build Tools needed**, everything runs in containers) using winget when it is available and falling back to a direct download of the official installers when it is not (so Windows LTSC / Server / locked-down images without the App Installer still work), reboots-and-resumes automatically when Windows requires it, generates `.env` secrets, prompts for an LLM provider API key (required — pass `-SkipLlmPrompt` on the underlying script to configure it later), then starts the fully containerized dev stack from `starters/docker/compose.fullapp.dev.yml`: the app (`:3000`, hot reload), the MCP server (`:3001`), the OpenCode agent (`:4096`), plus PostgreSQL, Redis, and Meilisearch. It waits for everything to become healthy and prints URLs and superadmin credentials.

- First boot takes 10–20 minutes (the container installs, builds, and seeds); progress is streamed from the build splash on `http://localhost:4000`. Subsequent starts take seconds.
- `starters\docker\windows\stop-windows.bat` stops the stack (data survives in Docker volumes).
- Re-running `start-windows.bat` is always safe — every step is idempotent. Useful flags on the underlying script (`starters\docker\windows\start-dev.ps1`): `-Status`, `-Logs`, `-Restart`, `-Reset` (deletes all data), `-DryRun`, `-NonInteractive`, `-SkipLlmPrompt`, `-Rebuild`, `-IncludeNativeToolchain`.
- **No administrator rights?** The launcher elevates *only* to install missing prerequisites. If a container runtime + WSL2 are already present (e.g. provisioned by IT), it detects that and runs the whole flow without admin. On a machine where you can't elevate at all, run `powershell starters\docker\windows\start-dev.ps1 -NoAdmin` — it skips every admin step (and downloads the repo as a ZIP when Git is missing) and, if the runtime/WSL2 aren't installed, tells you exactly what to ask IT for.
- **Rancher Desktop instead of Docker Desktop?** Fully supported — common where Docker Desktop licensing is not permitted. Double-click `start-windows-rancher.bat`: it auto-detects an existing Rancher Desktop (machine-wide or per-user install), and on a machine where WSL2 is already present it installs Rancher **per-user with no admin rights at all**. It must run the **dockerd (moby)** engine — the launcher requests this itself via `rdctl`. (The equivalent flag on the underlying script is `-Runtime rancher`.)
- **IDE setup:** edit the project folder with any Windows IDE (VS Code, JetBrains, Cursor) — hot reload flows through the bind mount, and WSL never enters your workflow (`/mnt/c/...` is only how a WSL shell would see the disk; don't run the project from there). One caveat: `node_modules` lives in a container volume, not on the host, so TypeScript IntelliSense/ESLint need one of: **(a)** VS Code → *Dev Containers: Attach to Running Container…* on the `app` container and open `/app` (recommended — full IntelliSense, zero host tooling), or **(b)** install Node 24 and run `yarn install` once, purely to feed the editor (the app still runs in containers). Power users who want maximum filesystem performance can use the full [WSL2-native guide](./wsl2) instead.
- **Blocked downloads / air-gapped?** Drop the official installers (Git for Windows, Docker Desktop or Rancher Desktop, `wsl_update_x64.msi`) into an `installers` folder in the repository root (or next to the launcher scripts) (or point `OM_INSTALLERS_DIR` at them) — the launcher validates and uses them without any network access. winget is never required.

The sections below cover the **native (non-Docker) toolchain** as an alternative for contributors who prefer running the app directly on Windows.

:::tip WSL2 recommended for full monorepo development
For the best developer experience on Windows — native Linux tooling, faster file I/O, full Docker support — use **WSL2 with Ubuntu** instead of running everything natively. See the dedicated [Windows with WSL2](./wsl2) guide which covers Ubuntu setup, memory configuration, Docker, GitHub CLI, and network bridging to a Windows-hosted database.

The steps below use native Windows tooling (PowerShell / cmd) and remain a valid alternative.
:::

## Prerequisites {#windows}

### 1. Git

Download the installer from [git-scm.com/download/win](https://git-scm.com/download/win). During setup choose **"Git from the command line and also from 3rd-party software"** when asked about PATH.

After installation, configure line endings to avoid CRLF issues in shell scripts:

```powershell
git config --global core.autocrlf input
```

Verify: `git --version`

### 2. Node.js 24

Download the **Windows Installer (.msi)** for Node.js 24 from [nodejs.org/en/download](https://nodejs.org/en/download). Run the installer with default settings.

After installation, **open a new PowerShell or Command Prompt window** so the updated `PATH` takes effect.

Verify: `node --version` → `v24.x.x`

### 3. Yarn 4.12.0

```powershell
corepack enable
corepack prepare yarn@4.12.0 --activate
```

:::warning PowerShell execution policy error
If you see this error when running `yarn`:

```
yarn.ps1 cannot be loaded because running scripts is disabled on this system.
```

This is a Windows PowerShell execution policy restriction. Fix it with a one-time user-scoped change:

```powershell
Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser
```

Then retry `corepack prepare yarn@4.12.0 --activate`.

**Alternative:** use **Git Bash** or **Command Prompt (cmd.exe)** instead of PowerShell — the execution policy restriction does not apply in those shells. All subsequent `yarn` commands in this guide work identically in cmd.exe or Git Bash.
:::

Verify: `yarn --version` → `4.12.0`

### 4. Visual Studio 2022 Build Tools

Required for native Node.js add-ons. Open **PowerShell as Administrator** and run:

```powershell
winget install Microsoft.VisualStudio.2022.BuildTools
winget install Microsoft.VCRedist.2015+.x64
```

If you prefer the graphical installer, make sure the **"Desktop development with C++"** workload is checked.

### Automated one-time setup (optional)

You can automate steps 3 and 4 by running the bundled helper script from an **Administrator PowerShell**:

```powershell
.\starters\hybrid\windows-toolchain.ps1
```

If Windows blocks the script (e.g. downloaded as part of a ZIP archive), unblock it first:

```powershell
Unblock-File .\starters\hybrid\windows-toolchain.ps1
```

The script validates and installs: WSL 2 defaults, Node.js 24, Yarn 4.12.0, Git PATH repair, VS 2022 Build Tools, and VC++ Redistributable. It does **not** install Docker Desktop, clone the repo, or run bootstrap commands.

## Infrastructure services

PostgreSQL is the most critical dependency — the app runs with a native PostgreSQL instance even without the rest of the Docker stack. Choose one option:

### Option A — Native PostgreSQL + pgAdmin (recommended)

The official Windows installer from **[postgresql.org/download/windows](https://www.postgresql.org/download/windows/)** bundles **pgAdmin 4**, a full-featured database GUI — no extra installation needed.

1. Download and run the EDB installer. Note the **port** (default `5432`) and the **`postgres` superuser password** you set.
2. Open pgAdmin 4 from the Start menu to browse and manage your databases visually.
3. Create the application database. In pgAdmin's Query Tool or PowerShell:

   ```powershell
   psql -U postgres -c "CREATE DATABASE \"open-mercato\";"
   ```

4. Set `DATABASE_URL` in `apps\mercato\.env` (after cloning):

   ```env
   DATABASE_URL=postgres://postgres:<your-password>@localhost:5432/open-mercato
   ```

For **Redis** and **Meilisearch**, start only those two services via Docker (requires Docker Desktop):

```powershell
docker compose --project-directory . -f starters/docker/compose.infra.yml up -d redis meilisearch
```

Most development features work without Meilisearch — you can skip it and add it later.

### Option B — Docker Desktop (all infrastructure)

Install [Docker Desktop for Windows](https://docs.docker.com/desktop/setup/install/windows-install/) with the **WSL 2 backend** enabled (Settings → General → "Use the WSL 2 based engine").

Start all infrastructure from the repository root:

```powershell
yarn infra:up
```

## Microsoft Defender exclusion (recommended)

Without a Defender exclusion, `yarn install` and file-watching can be significantly slower. Run once in an elevated PowerShell:

Start-Process powershell -Verb RunAs -ArgumentList "-NoProfile -ExecutionPolicy Bypass -Command `"Add-MpPreference -ExclusionPath '$((Get-Location).Path)'`""

## Development runtime reference

| Command | Description |
|---------|-------------|
| `yarn dev` | Compact runtime — splash at `http://localhost:4000`, app at `http://localhost:3000/backend` |
| `yarn dev:verbose` | Same runtime with raw passthrough logs |
| `yarn dev:greenfield` | Full fresh boot: build → generate → reinstall → dev |
| `yarn dev:ephemeral` | Throwaway database on a random port (requires Docker) |
| `yarn dev:classic` | Legacy mode — disables splash, raw output |

Press `d` while `yarn dev` is running to toggle raw log output. If a stage fails, the runner automatically expands and prints the raw error.

### Run multiple persistent local instances against the same PostgreSQL server

`yarn dev`, `yarn dev:greenfield`, and `yarn dev:app` accept an optional `--database-name[=<name>]` flag that rewrites the database segment of `DATABASE_URL` in `apps/mercato/.env`. Without the flag, behavior is unchanged.

```bash
# explicit name; you'll be asked once whether to persist .env (default yes)
yarn dev:greenfield --database-name=pricing_v2

# bare flag derives the database name from the current working directory
yarn dev --database-name

# one-off review run that does not edit .env
yarn dev --database-name=review_1720 --no-update-env
```

The override only touches the pathname segment of `DATABASE_URL`, so credentials, host, port, schema (`?schema=…`), and other query parameters are preserved. CI / non-interactive runs default to updating `.env`; pass `--no-update-env` to opt out, or set `OM_DEV_DATABASE_UPDATE_ENV=false`.
