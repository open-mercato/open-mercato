# Payment Link Pages

The `payment_link_pages` module owns the hosted public `/pay/[token]` route. It is intentionally separate from `payment_gateways` so apps can replace or eject the public page without removing gateway adapters, transaction APIs, or webhook handling.

## Architecture

- `payment_gateways` creates payment links and stores `GatewayPaymentLink` / `GatewayTransaction`.
- `payment_link_pages` renders the public page and exposes a dedicated page-data API.
- Gateway provider packages still inject provider checkout widgets through `payment-gateways.payment-link:<providerKey>`.

## Stable UMES surfaces

### Component replacement

- Page handle: `page:/pay/[token]`
- Section handles:
  - `section:payment_link_pages.brand`
  - `section:payment_link_pages.summary`
  - `section:payment_link_pages.checkout`

Use wrapper/props overrides first. Use full replacement only when you need a different page layout.

#### Replacing a section from your app module

Create `widgets/components.ts` in your module and target a section handle:

```typescript
// apps/mercato/src/modules/my_checkout/widgets/components.ts
import type { ComponentOverride } from '@open-mercato/shared/modules/widgets/component-registry'
import { buildPaymentLinkPageSectionHandle } from '@open-mercato/shared/modules/payment_link_pages/types'

export const componentOverrides: ComponentOverride[] = [
  {
    target: { componentId: buildPaymentLinkPageSectionHandle('brand') },
    priority: 50,
    metadata: { module: 'my_checkout' },
    replacement: (await import('./CustomBrandSection')).default,
  },
]
```

The replacement component receives the same props as the default ŌĆö `data`, `loading`, `error`, `locale`:

```tsx
// apps/mercato/src/modules/my_checkout/widgets/CustomBrandSection.tsx
'use client'

export default function CustomBrandSection({ data, loading }: {
  data: { link?: { metadata?: Record<string, unknown> } } | null
  loading: boolean
}) {
  const brandName = (data?.link?.metadata?.brandName as string) ?? 'My Store'
  return (
    <div className="flex items-center gap-3">
      <span className="text-xl font-bold">{brandName}</span>
    </div>
  )
}
```

#### Wrapping the entire page layout

To wrap the page root (change layout, add a banner, etc.) without fully replacing it:

```typescript
// widgets/components.ts
import { PAYMENT_LINK_PAGE_COMPONENT_HANDLE } from '@open-mercato/shared/modules/payment_link_pages/types'

export const componentOverrides: ComponentOverride[] = [
  {
    target: { componentId: PAYMENT_LINK_PAGE_COMPONENT_HANDLE },
    priority: 50,
    metadata: { module: 'my_checkout' },
    wrapper: (Original) => {
      const WrappedPage = (props: object) => (
        <div className="my-custom-theme">
          <Original {...props} />
          <footer className="text-center text-sm py-4">Powered by My Store</footer>
        </div>
      )
      WrappedPage.displayName = 'MyCheckoutPageWrapper'
      return WrappedPage
    },
  },
]
```

Run `npm run modules:prepare` after adding or modifying `widgets/components.ts`.

#### Available section prop types

| Handle | Props type | Key fields |
|--------|-----------|------------|
| `section:payment_link_pages.brand` | `BrandSectionProps` | `data`, `loading`, `error`, `locale` |
| `section:payment_link_pages.summary` | `SummarySectionProps` | `data`, `loading`, `error`, `locale` |
| `section:payment_link_pages.checkout` | `CheckoutSectionProps` | `data`, `loading`, `error`, `locale`, `password`, `unlocking`, `customerForm`, `customerSubmitting`, `customerError`, `customerTermsAccepted`, `onPasswordChange`, `onUnlock`, `onCustomerFieldChange`, `onCustomerTermsAcceptedChange`, `onSubmitCustomer`, `onRefreshLink` |
| `page:/pay/[token]` | `RootProps` | `data`, `loading`, `error`, `customCss`, `beforeContent`, `heroContent`, `summaryContent`, `checkoutContent`, `afterContent` |

### Injection spots

- `payment-link-pages.pay:before`
- `payment-link-pages.pay:hero`
- `payment-link-pages.pay:summary`
- `payment-link-pages.pay:checkout`
- `payment-link-pages.pay:after`

These spots receive the current page payload under `data` and `context`, including:

- `paymentLink`
- `transaction`
- `metadata`
- `customFields`
- `customerCapture`
- `enrichedBy`

#### Injecting a widget from your app module

Create an injection widget and map it to a spot:

```tsx
// apps/mercato/src/modules/my_checkout/widgets/injection/trust-badge/Widget.tsx
'use client'

export default function TrustBadgeWidget({ context }: {
  context: { paymentLink: { amount?: number; currencyCode?: string } | null }
}) {
  return (
    <div className="flex items-center gap-2 rounded-lg border p-3 text-sm">
      <span>Secured by SSL</span>
      <span>Money-back guarantee</span>
    </div>
  )
}
```

```typescript
// apps/mercato/src/modules/my_checkout/widgets/injection/trust-badge/widget.meta.ts
export const metadata = { id: 'my-checkout-trust-badge', label: 'Trust Badge' }
```

```typescript
// apps/mercato/src/modules/my_checkout/widgets/injection-table.ts
export default [
  { widgetPath: 'trust-badge', spotId: 'payment-link-pages.pay:summary' },
]
```

### Enrichers

The public page API is `GET /api/payment_link_pages/pay/[token]`.

Target entity for response enrichers:

```text
payment_link_pages.payment_link_page
```

Enrichers must remain additive-only and should namespace their additions under `_<module>`.

### Events

The module emits:

- `payment_link_pages.page.viewed`
- `payment_link_pages.page.unlocked`

Use these for notifications, workflow triggers, analytics side effects, or link-distribution follow-ups.

## JSON metadata

The create-transaction API accepts arbitrary JSON for the public page:

```json
{
  "paymentLink": {
    "enabled": true,
    "title": "Invoice INV-10024",
    "metadata": {
      "brandName": "Acme Commerce",
      "logoUrl": "https://merchant.example.com/logo.svg",
      "customCss": ".payment-link-accent { color: #0f766e; }"
    }
  }
}
```

That metadata is returned to the page under `link.metadata`.

The default host already reads a few optional keys:

- `brandName`
- `logoUrl`
- `securitySubtitle`
- `customCss`

Everything else is left available for custom replacements, injections, or app-specific ejected pages.

## Custom public URL

When creating a pay link you can optionally choose the final public path segment instead of using a random token.

```json
{
  "paymentLink": {
    "enabled": true,
    "title": "Invoice INV-10024",
    "token": "invoice-inv-10024"
  }
}
```

Rules:

- optional; when omitted, Open Mercato generates a random secure token
- must be unique per tenant and organization
- must use only lowercase letters, numbers, and dashes
- becomes the last segment of `/pay/<token>`

## Optional customer capture

Pay links can optionally act like a lightweight checkout and collect customer data before the gateway widget is shown.

Create-session payload:

```json
{
  "paymentLink": {
    "enabled": true,
    "title": "Invoice INV-10024",
    "customerCapture": {
      "enabled": true,
      "companyRequired": false
    }
  }
}
```

When enabled:

- the hosted page blocks the payment widget until the customer form is submitted
- if terms/GDPR markdown is configured, the hosted page blocks the customer form until the checkbox is accepted
- the public API returns the state under `link.customerCapture`
- the hosted page submits to `POST /api/payment_link_pages/pay/[token]/customer`
- Open Mercato reuses existing customer records when it can, and creates missing company/person records when it cannot

The default hosted form collects:

- company name (optional unless `companyRequired` is true)
- first name
- last name
- email
- phone

After submission, the stored state includes the matched/created entity ids and a small snapshot (`companyEntityId`, `personEntityId`, `companyName`, `personName`, `email`, `collectedAt`), which is returned to the page and available to UMES injections, replacements, and enrichers.

Terms / GDPR example:

```json
{
  "paymentLink": {
    "enabled": true,
    "title": "Invoice INV-10024",
    "customerCapture": {
      "enabled": true,
      "termsRequired": true,
      "termsMarkdown": "## GDPR consent\n\nI consent to processing my personal data for payment handling."
    }
  }
}
```

The markdown content is returned under `link.customerCapture.termsMarkdown`, and the acceptance state is reflected in `link.customerCapture.termsAcceptedAt`.

## Custom fields and fieldsets

Pay-link page metadata can also be collected with the custom-fields engine.

Entity id:

```text
payment_link_pages:payment_link_page
```

Recommended flow:

1. Define fieldsets and custom fields for `payment_link_pages:payment_link_page`.
2. In the create transaction dialog, select the desired fieldset.
3. Fill the generated custom fields.
4. Open Mercato stores:
   - `paymentLink.customFieldsetCode`
   - `paymentLink.customFields`
5. The public page API returns them as:
   - `link.customFieldsetCode`
   - `link.customFields`

This makes it practical to maintain multiple predefined page variants, for example invoice links, event-payment links, or pickup-payment links.

You can combine this with JSON metadata and pay-link custom fields. A common pattern is:

1. define a page fieldset for a specific pay-link variant
2. enable `customerCapture`
3. use custom fields for page-specific metadata
4. use `link.customerCapture`, `link.metadata`, and `link.customFields` together inside custom page components

## Ejecting the page

If you want to build the page entirely from scratch:

1. Remove `{ id: 'payment_link_pages', from: '@open-mercato/core' }` from your app `src/modules.ts`.
2. Create an app module that owns `frontend/pay/[token]/page.tsx`.
3. Keep `payment_gateways` enabled so session creation, providers, and webhooks continue to work.
4. Read from `GET /api/payment_link_pages/pay/[token]` or build your own host against the gateway data model.

This keeps the payment infrastructure intact while letting the public page diverge completely.
